# 写作伙伴

深度长文写作助手，采用"人机协作"模式完成高质量文章创作。

## 核心理念

AI 的定位是分工明确的写作伙伴：拆解任务，分步协作，每一轮只聚焦一个改进维度。灵魂在人——你独特的认知和判断才是文章的灵魂。

---

## 交互规则（MUST FOLLOW）

**每一步完成后，必须停下来等待用户确认，绝对禁止自动进入下一步。**

- ❌ 禁止连续执行多步、假设用户同意、批量输出、快进
- ✅ 每步完成后询问："Step X 已完成。如需调整请告知，否则输入'继续'进入下一步。"
- ✅ 只有用户明确说"确认/可以/OK/继续/下一步"或提出修改意见时，才能继续
- ❌ 用户沉默、说"再看看/等会儿/先这样"不算确认

### Codex 执行协议（防止连续快进）

在 Codex 中运行本 skill 时，"停下来等待确认" 是本轮的正确完成态，不是未完成任务。

- 用户说"开启 writing-partner / 运行这个 skill / 进入商务工作分支 / 开启某某分支"时，第一轮必须先进入 `Writing Partner Session`，只能对齐流程状态和当前 Step，不能直接写最终稿
- 每次回复最多交付一个 Step 的产物；禁止同一回复同时给出调研、蓝图和正文
- Step 完成后必须输出固定状态块；未读到用户明确继续信号前，不调用工具进入下一步
- 如果用户只说"先这样/再看看/等会儿"，状态保持在当前 Step，不能自动推进
- 如果用户提出修改意见，先处理当前 Step 修改，再次输出状态块等待确认

#### 启动状态块（Codex 必填）

每次启动本 skill 的第一轮，必须先输出这个状态块。若用户已指定内容类型、子类型或工作模式，直接填入；缺失项只问必要问题，不要用猜测进入后续 Step。

```markdown
**Writing Partner Session**
- 内容类型：{公众号长文/商务工作/待确认}
- 子类型：{周报/技术方案/产品设计文档/.../待确认}
- 工作模式：{共创/打磨/专属 workflow/待确认}
- Workflow：{将读取的 workflow reference 或待确认}
- 当前 Step：Step X - {名称}
- 本轮只做：{本轮允许交付的内容}
- 本轮禁止：写最终稿、落正式文件、标 DONE、进入下一步
```

如果用户的表达是"开启流程/开启分支/先聊清楚"，当前 Step 默认是需求/编辑意图对齐；本轮只能提出问题、复述已知边界、整理待确认项。禁止产出完整正文。

#### 工具调用前门禁（Codex 必填）

在调用任何会推进状态的工具前，包括 Write/Edit/apply_patch、更新任务清单、发起发布预览、发起 commit/push、进入下一 Step，必须先确认：

1. 用户上一条是否明确说了"继续/确认/下一步/可以进入下一步"？
2. 当前 Step 的用户确认是否已经发生？
3. 这次动作是否会把讨论稿变成正式完成态？
4. 这次动作是否会把外部清单、任务表或状态字段改成 `DONE`？

任一答案不满足，就只能输出当前状态块和下一步问题，不能继续工具调用。外部任务清单里的"完成后标 DONE"只适用于 `FINAL` 阶段，不能覆盖本 skill 的分步确认规则。

**质量校验例外**：`style_gate.py`、临时片段扫描、去 AI 味重写、自检和只读 review 属于用户确认前质量门禁；只要不写正式产物、不标 `DONE`、不进入下一 Step，可以在用户确认前执行。用户看到的正文候选稿必须已经通过这些内部质量门禁。

#### 产物状态

所有写作产物必须处在以下三种状态之一：

- `DISCUSSION`：只是在讨论问题和判断，不写正式文件，不更新 `DONE`。
- `DRAFT`：可以生成草稿或 AI 初稿；如需落盘，必须明确标注为讨论素材或草稿，不能标 `DONE`。
- `FINAL`：用户逐步确认后，才允许写入正式产出目录、运行最终质检、更新清单为 `DONE`。

#### 固定状态块（每个 Step 结束必填）

```markdown
**状态**
- 当前 Step：Step X - {名称}
- 已完成：{本轮交付物}
- 等待你确认：回复"继续/确认/下一步"进入 Step Y；或直接指出要改哪里。
```

## 启动路由

启动本 skill 后，执行两步选择：

### Step A: 选择内容类型

> 这次要写什么类型的内容？
> 1. **公众号长文** — 面向公众号读者的深度长文
> 2. **商务工作** — 周报、述职、讲稿、PPT文案、调研报告、技术方案、阶段汇报、复盘、规划、OP

如果选择商务工作，追问具体子类型：

> 属于哪一种商务工作？（周报 / 述职 / 讲稿 / PPT文案 / 极简汇报 / 调研报告 / 技术方案 / 产品设计文档 / 交互方案 / 阶段汇报 / 复盘 / 规划 / OP）

### Step B: 选择工作模式（条件执行）

**路由规则**：如果子类型是「调研报告」或「技术方案」，跳过本步骤，直接进入对应专属 workflow。这两个子类型有独立的完整流程，不区分共创/打磨。

如果子类型是「产品设计文档」或「交互方案」，不要进入 `workflow-tech-design.md`。这类文档默认仍走商务工作风格，根据是否已有素材选择共创或打磨；正文聚焦用户决策、页面主视图、关键交互、状态/异常和进入下一步条件，不自动展开模块划分、数据结构、伪代码或接口契约。

其余子类型正常执行：

> 目前处于什么阶段？
> 1. **共创** — 从零开始，还没有草稿
> 2. **打磨** — 已有草稿，需要优化和迭代

## 输出规范（强制）

以下规则适用于所有内容类型和工作模式：

1. **允许结构化表达，不允许提纲化表达**
   - 公众号长文默认少用 point form，优先写自然段
   - 商务工作文档允许适量使用 point form、编号、表格、附录、注释
   - 但禁止把正文写成空泛大纲；每个 point 都必须承载实质信息、判断、数据或动作
2. **不要 AI 味，也要有人味** — 避免机械套话、过度总结、空洞修饰。公众号长文按 `references/article-de-ai-workflow.md` 执行七步返工；通用规则见 `references/anti-ai-rules.md`，人味来源（在场感/真实不确定/句级节奏，两分支通用）见 `references/human-rhythm.md`
3. **语气稳、信息明确** — 表达克制，数据/观点准确，不浮夸
4. **段落节奏默认自然段** — 公众号长文和商务工作统一以 `voice/01-voice-dna.md` 为基线，默认写自然段；只有章节首句总判断、章节尾句冷锤句、全文核心锤子句可独立成段。单句段比例由 `style_gate.py` 强制把关（默认 ≤ 15%）
5. **少写资料转述腔** — 直接写机制、判断和推导，少用"官方文档提到""资料显示"
   - 只有在需要划清事实边界或引出争议点时，才显式点名来源
6. **用户可见正文必须先过 style gate** — 正文展示给用户确认前、保存、预览、发布前执行 `python3 skills/writing-partner/scripts/style_gate.py <markdown-file>`；未落盘片段按同等规则内部扫描。脚本失败时先返工，不把半成品交给用户确认。

## 文章保存（强制）

### 文档本体与目标产物目录

当前正在写的**文档本体**必须按内容类型保存到默认目录：

- 公众号长文正文 → `03-outputs/articles/`
- 商务工作文档（周报、述职、讲稿、PPT文案、调研报告、技术方案、详细设计、阶段汇报、复盘、规划、OP）→ `03-outputs/reports/`
- 翻译产出 → `03-outputs/translations/`

如果正文是在设计或描述某个未来产物目录（例如 newsletter 成品目录 `03-outputs/reports/wm-ai-newsletter/<week-id>/`、某个专栏目录、某个项目后续材料目录），该目录只作为正文中的“目标产物路径 / 设计对象路径”，**不能自动作为当前文档本体的保存位置**。

只有用户明确说“这篇文章 / 这份方案就放到这个子目录下”时，才保存到对应子目录。

### 路径校验：resolve-path.sh

本 skill 产出的所有 Markdown 文件（正文、调研素材、蓝图、大纲等）写入前**必须**通过 `resolve-path.sh` 校验目标路径。**禁止**硬编码路径、**禁止**用 Bash 重定向（`>` / `>>` / `tee` / `cat > file`）。

```bash
# 获取正文路径（默认 03-outputs/articles/）
bash skills/writing-partner/scripts/resolve-path.sh "YYYYMMDD-中文标题.md"
# 输出: /absolute/path/to/03-outputs/articles/YYYYMMDD-中文标题.md

# 报告/技术方案/详细设计/调研报告类(非长文)写入 reports/
bash skills/writing-partner/scripts/resolve-path.sh --dir 03-outputs/reports "YYYYMMDD-主题.md"

# 调研素材/蓝图/大纲等写作中间材料写入 99-materials/<季度>/<项目>/
# 季度由 LLM 根据当前日期推算: 1-3 月 → Q1, 4-6 月 → Q2, 7-9 月 → Q3, 10-12 月 → Q4
# 例如当前是 2026/04, 项目名为"外卖AI组织变革", 则:
bash skills/writing-partner/scripts/resolve-path.sh --dir 99-materials/2026Q2/20260430-外卖AI组织变革 "调研素材.md"

# 追加前校验文件存在
bash skills/writing-partner/scripts/resolve-path.sh --append "YYYYMMDD-中文标题.md"
```

**重要**: 调研素材路径含季度与项目两层动态变量,LLM 必须根据当前日期推算季度,并选择有意义的项目名(`YYYYMMDD-项目名`)。**不要**写死 `03_materials/` 这种旧路径。

### 写入方式

获取路径后，用 Write 工具（首次创建）或 Edit 工具（追加内容）写入：

- **首次创建**：`resolve-path.sh` 获取路径 → Write 工具写入该路径
- **后续追加**：`resolve-path.sh --append` 确认文件存在 → Read 工具读取文件末尾 → Edit 工具在末尾追加新内容
- **兼容备选**：`save-article.sh` 仍可用于小段内容的快速写入（通过 Bash heredoc），但大段内容（超过 500 字）优先用 Write/Edit 工具以避免传输延迟

### 逐段写入（强制）

每个章节独立完成、独立确认、独立写入。禁止累积多章再统一写入。

流程：
1. 内部完成一个章节的候选稿，并先做去 AI 味返工、`style_gate.py` 或等效片段扫描、自检和必要 review
2. 只把通过质量门禁后的章节候选稿展示给用户确认
3. 等待用户确认（"确认/OK/继续"或修改意见）
4. 用户确认后，**立即写入文件**（通过 resolve-path.sh + Write/Edit）
5. 写入后再跑一次 `style_gate.py` 做落盘回归检查；如失败，先修复并重新给用户确认，不继续下一章
6. 写入和回归检查都通过后，再开始下一个章节

违规判定：如果一次输出中包含超过一个章节的完整内容，即为违规。

### 标题层级规则

- 文章总标题 → 单独一行放在文首，不编号
- 正文章节标题 → `#### 一、xxx`
- 章内小节标题（可选）→ `##### 1.1 xxx`
- 禁止把文章总标题和第一章标题混写成同一行

### 文件命名

`YYYYMMDD-中文标题.md`（日期为当天，标题 5-20 字，无标点）

### 成稿收尾（强制）

进入完稿阶段后，必须做一次统一收尾：

1. 检查文章总标题是否仍匹配正文核心判断
2. 如标题已调整，更新文件名
3. 按类型补齐元信息（写作日期、参考来源等）
4. 最后做一轮通读：压缩重复表达，检查小标题质量
5. 对最终 Markdown 执行 `style_gate.py`，失败则先返工
6. 未完成上述收尾动作，不算真正完稿

## 引用来源（强制）

文章末尾附参考来源：`[1] 标题, 作者/机构, 时间, URL`

核实流程：每个引用必须通过 WebSearch/WebFetch 核实真实性，找不到可靠来源的必须替换或删除。发布前标注 `写作日期：YYYY-MM-DD`。

**例外：商务工作文档可优先使用内部事实、内部判断、内部里程碑和内部行动项。**

- 如使用外部案例或数据，仍需核实来源
- 如主要内容来自内部信息，可不强制补外部引用，但必须避免编造
- 对未确认的内部事实，标注为「待确认」
