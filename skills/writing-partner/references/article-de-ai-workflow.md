# 公众号长文去 AI 味七步路径

本文件用于公众号长文分支的正文输出前返工。它不是完稿润色，而是每节成文前必须经过的内部改写路径。

目标：让正文像一个自己踩过这些坑、见过真实账本的人在给判断，而不是 AI 在做正确总结。去 AI 味（消毒）和补人味（在场证据）在同一条路径里完成，不把人味留到质检阶段补妆。

---

## 触发时机

- 每节正文输出给用户前
- 打磨模式中每段改写输出前
- `style_gate.py` 命中 A/B 纠偏句、说教转场、空泛结尾时
- 质检发现"AI 味太重"但不知道如何返工时

**前置加载（强制）**：执行本路径前，必须读取 `style-examples.md` 的"公众号长文示例"节——正反例对照是比规则清单更强的校准源，不能只按条文扫描。

用户默认只看到返工后的正文。除非用户要求，不展示内部检查表。

---

## 七步返工

### 1. 事实密度

检查每个抽象判断后面有没有至少一种硬支撑：

- 具体对象：公司、产品、团队、角色、系统、流程
- 具体动作：谁做了什么，怎么做，做到哪一步
- 具体数字：时间、成本、人力、比例、频次、规模
- 具体约束：预算、权限、组织惯性、技术边界、迁移成本
- 具体反例：为什么常见做法会失败

补不出支撑时，不要编。标注为待补事实，回到调研或向用户要材料。

### 2. 作者在场证据

检查本节有没有素材池（`99-materials/<项目>/一手素材.md`）可消耗的亲历落点：

- **有料**：以无人称亲历级细节织入——当时的误判、发现问题的时点、返工次数、卡住的具体环节（"demo 两周就出来了，上线用了三个月"）。标注"亲手做/亲眼看"的条目可写过程颗粒；"听说·二手"只写结果事实。
- **本节是"亲自做过一件事"**：按过程叙事弧改写——原始想法 → 在业务链路里卡住 → 改了一版 → AI 接进去 → 复盘发现新问题；按发现顺序推进，不按结论结构倒叙。
- **无料**：确认本节保持纯推理是合理的，或标注"在场感素材缺失"回头向用户要；禁止编造细节。
- 真实不确定只落在素材池"没把握"条目对应的判断上（三类边界：归因/未来/边界）；没有对应条目就不写。
- 全程无人称。在场感来自细节颗粒度，不来自"我"。

### 3. 作者判断

把资料转述改成作者判断：

- 不写："资料显示，很多企业正在关注 AI Agent。"
- 改写："企业真正该关注的不是 Agent 这个词，而是它能不能把一个岗位的重复动作吃掉。"

判断要落在当前问题上，不要泛化成行业常识。公众号长文禁用"我觉得""我认为"等第一人称判断前缀，用无人称冷静断言。

### 4. 约束与代价

每个建议后面至少补一个约束或代价：

- 做这件事最容易卡在哪里？
- 哪种团队不适合照抄？
- 如果不做，代价是什么？
- 如果做错，最可能浪费什么？

没有约束的建议像广告词；有约束，才像真实负责人说话。

### 5. 声口校准与句级节奏

按 `voice/01-voice-dna.md` 校准段落节奏和句式，叠加 `content-article.md` 的公众号专属增量：

- 段落以自然段为主（继承 `voice/01` 的"三段式推进"基线），只有章节首句总判断、章节尾句冷锤句、全文核心锤子句允许独立成段
- 章节用硬编号（一、二、三...），章节之间可以硬切，不用温柔过渡
- **句级把关**：平实、信息密度高的句子是声口本体，不为节奏制造表演性短拍句（"人就往后退"式）、提炼鼓点句（"问题写出来就一句话"式）或伪口语替换（"脑子里"换"经验里"）——2026-07-09 A/B 盲评实证这些是 AI 味；长句只在信息推进停滞时返工，方式是删冗余或拆信息（顿号列举收束句是刻意密度收拢，保留）
- 用账本、真相、反例、顺序清单推进（详见 `content-article.md` 第六章 ROI 账本、第七章敢说真话、第八章反升华结尾）
- 禁止朋友式亲近、情绪铺垫、文化升华
- 允许具体对比，但对比两边必须具体，后面必须跟判断

### 6. 门禁扫描

对照 `anti-ai-rules.md` 和 `style_gate.py` 扫描：

- A/B 模板纠偏句："不是 A，而是 B"
- 递进升格句："不仅是 A，更是 B"
- 说教转场："有一个很关键""关键在于""每一步都不神秘"
- 万能总结："这意味着""本质上""值得注意的是"
- 空泛结尾："让我们一起期待""未来可期"
- 作者意图说明："我更关心的是""我不太想从...讲起""本文想表达"
- 零信息桥接："这段话没有错，但...""放到 X 场景里会更具体"
- 专家式标题："X 卡住，往往卡在 Y""关键在于 X"
- 未定义伪产品名：临时创造 `X Agent`、`X 大脑`、`X 中台`

命中后先改写，再输出。不要把"后面再润色"留到完稿阶段。

### 6.5 探讨口吻检查

公众号长文如果目标是"探讨"，标题和段落就要从具体问题进入，不要用专家讲课式判断压读者。

输出每一节前，额外问四个问题：

- 标题是不是一个具体问题、业务场景或动作链？如果像"AI 卡住，往往卡在..."，先改。
- 这一句是不是只在解释作者写法？如"我更关心的是""我不太想从..."，删掉后直接写内容。
- 这一句是不是只在转场？如"这段话没有错，但..."，删掉后下一段能否自然接上。
- 有没有临时造 `X Agent`、`X 大脑`、`X 中台`？如果不是已有概念，改成业务对象、系统动作或人审链路。

例子：

- 不写："我更关心的是，AI 进入业务以后，能不能少一点'看起来懂'。"
- 改写："AI 进入业务以后，光是'看起来懂'不够。它看到一个指标，能不能追到具体对象。"

### 7. 分类返工

发现 AI 味后，按问题类型直接采取动作：

| 问题类型 | 返工动作 |
|---|---|
| 空泛总结 | 补具体对象、动作、数字、边界；补不出就删 |
| 教科书科普 | 把知识点贴回当前问题，写它解释了什么、改变了什么判断 |
| 模板对照 | 删掉"不是 A，而是 B"，改成具体两边对比 + 作者判断 |
| 资料转述 | 改成机制、影响、取舍和结论 |
| 伪亲近 | 删"我也面临这个问题"，改成二人称场景或冷判断 |
| 升华癖 | 改成顺序清单、冷判断回扣或类比定调 |
| 漂亮废话 | 问"删掉会不会损失信息"；不会就删 |
| 专家教育标题 | 改成具体业务问题、对象或动作链 |
| 作者意图说明 | 删掉意图说明，直接写判断和依据 |
| 未定义伪产品名 | 改成业务对象或系统动作；确需保留时先定义 |
| 缺在场感 | 细节全是转述级，看不出做过；从素材池取亲历条目补时点、误判、返工次数；无料显式降级 |
| 句级表演腔 | 为节奏造的短拍句、提炼鼓点句、伪口语；换回平实的信息句和操作句，节奏问题靠删冗余不靠造短句 |
| 过程倒叙成报告 | 亲历章节按结论结构写；按发现顺序重排：原始想法→卡住→改版→接入→复盘 |

## 输出前自查

- [ ] 每个核心判断都有事实、动作、边界、数字或反例支撑
- [ ] 素材池有料时本节已消耗亲历级细节；无料时已显式降级，没有编造
- [ ] 真实不确定只落在素材池"没把握"条目对应的判断上，没有口癖化
- [ ] 资料没有停留在"资料显示"，已经转成作者判断
- [ ] 建议写清了约束、代价或不适用场景
- [ ] 公众号长文没有朋友式亲近、情绪铺垫和文化升华；全程无人称
- [ ] 模板化纠偏句和说教转场已清零
- [ ] 标题没有专家教育口吻，正文没有作者意图说明和零信息桥接句
- [ ] 没有临时创造未定义的 `X Agent`、`X 大脑`、`X 中台`
- [ ] 没有为节奏制造的短拍句、提炼鼓点句或伪口语；长句只在信息推进停滞时才返工
- [ ] 结尾落在顺序清单、冷判断或类比定调

通过后再把正文展示给用户。
