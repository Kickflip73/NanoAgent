# 反 AI 痕迹规则

本文件汇总所有内容类型共用的反 AI 写作规则，以及按类型细分的专项规则。在写作和质检阶段按需加载。

---

## 零、默认目标

所有通过 writing-partner 产出的写作文档，都默认执行去 AI 味处理。这不是完稿后的可选润色，而是起草、改写、质检三个阶段都要遵守的基础约束。

去 AI 味的目标：文本要像一个真实负责人、研究者或作者在表达判断。

- 有具体事实，不用抽象套话撑篇幅
- 有真实取舍，不把问题写成无代价的完美方案
- 有个人判断，不只做资料复述
- 有业务对象、动作、约束和下一步，不停留在概念归纳
- 有必要的不确定性，不为了完整感编造结论

如果一段文字删掉后不影响全文信息量，优先删除；如果一段文字只是在重复“方向正确、仍需完善、持续推进”，必须改成具体判断、具体动作或具体待决策问题。

## 一、通用禁用词（跨类型）

以下词汇/句式一旦出现，必须替换：

| 禁用 | 替换建议 |
|------|----------|
| 综上所述 / 总的来说 | 用具体的回扣句，直接说核心结论 |
| 值得注意的是 / 不难发现 | 删掉，直接说后面的内容 |
| 首先...其次...最后 | 删掉，直接按逻辑顺序写；跳跃时补论证链或章节硬切，不补转场句 |
| 让我们来看看... / 接下来让我们... | 删掉，直接进入内容 |
| 在当今...的时代 / 随着...的发展 | 从具体事件或事实切入 |
| 说白了 | 坦率的讲、其实就是 |
| 意味着什么 / 这意味着 | 那结果会怎样呢、所以呢 |
| 本质上 | 说到底、其实 |
| 换句话说 | 你想想看、也就是说 |
| 不可否认 | 删掉，换成正面陈述 |

### 二元对照句式默认禁用

以下句式会让正文显得像模板化纠偏。用户可见正文默认清零；用户明确要求保留某一句时，必须在质检报告里列出保留原因。

| 慎用句式 | 问题 | 替换方向 |
|----------|------|----------|
| 不是 A，而是 B / 不是 A，是 B / 是 A，而不是 B | 模板化纠偏感强 | 直接写 B 的判断和依据 |
| 不仅是 A，更是 B | 容易制造虚假的递进感 | 拆成两个独立事实，保留真正更重要的那个 |
| 值得注意的是 | 常见 AI 转场 | 直接写后面的判断 |
| 但不能 / 但不应 / 不能...而要... | 容易变成“先肯定后刹车”的机械结构 | 写清边界条件或失败代价 |
| 一方面 / 另一方面 | 容易变成平衡腔 | 如果有取舍，直接写推荐方向 |
| 更重要的是 | 高频使用会显得每点都重要 | 只在真正优先级最高时保留 |

默认处理：改成直接判断、因果句、边界句或具体动作句。确需保留时，必须满足两个条件：这句话在纠正关键误解；删掉后会损失判断精度。

注意：这里禁用的是模板化纠偏句，不是禁止所有对比。公众号长文允许人物、时间、方案、表里等实质对比；对比两边必须具体，后面必须跟作者判断或行动落点，不能写成空泛的“不是 A，而是 B”模板。

## 二、硬门禁脚本

所有 writing-partner 用户可见正文，在展示、保存、预览或发布前必须运行：

```bash
python3 skills/writing-partner/scripts/style_gate.py <markdown-file>
```

如果正文还没有落盘，先对即将输出的片段做同等规则的内部扫描；一旦落盘，立即跑脚本。脚本返回非 0 时，不输出“待后续润色”，先按脚本提示返工。

### 高频说教句式硬扫描

以下句式一旦出现，默认重写。它们的问题不是单个词难听，而是会把作者姿态推成"老师在讲道理"，读者会明显感到 AI 味。

| 禁用 / 慎用 | 问题 | 替换方向 |
|-------------|------|----------|
| 有一个很关键 / 有一点很关键 | 先摆姿态，再给判断，像讲课 | 直接写判断和影响 |
| 关键在于 / 真正重要的是 | 容易变成万能转场 | 如果确实关键，写清影响哪个结果 |
| 这里面每一步都不神秘 / 并不复杂 / 说穿了 | 轻视真实执行难度，说教感强 | 改成具体步骤、难点和边界 |
| 这背后反映的是 / 背后的逻辑是 | 把具体问题抽象化，容易空转 | 先写事实，再写它改变了什么判断 |
| 可以看到 / 不难看出 | AI 常用观察句，信息密度低 | 删掉，直接写结论 |
| 从 X 到 Y，本质是 Z | 结构漂亮但常常虚 | 拆开写真实变化链条：X 怎么变成 Y，Z 影响什么动作 |
| 撞到天花板 / 已经吃不消 / 撑不下去 | 夸大问题严重度，AI 味浓的纠偏开场 | 写具体卡点和发生场景，让读者自己判断严重度 |
| 这几件事推着 X 往 Y 走 / 必然走向 X / 趋势倒逼 X | 拟人化"推动力"叙事，把判断包装成命运 | 直接陈述判断、列依据，把因果链写出来 |
| 必须升级到 X / 只能选择 X / 别无他法 | 绝对化强迫语气，把"主动选 X"包装成"被迫选 X" | 写"选 X 的几条理由"，承认还有其他选项但本次为何不选 |
| 下面分 N 节展开 / 第一节...第二节... / 下文将... / 接下来会讨论... | 章节标题已经回答了"接下来要讲什么"，再写一段预告是目录腔 | 直接进入第一个小节，不写章节预告段 |
| 本章主要包括以下几个方面 / 本节将从以下几个维度展开 | 同上，是 PPT 大纲腔挪到正文 | 删掉，让小节标题自己说话 |
| 这一章的小结 / 本章小结 / 小结一下 / 综上本章 | 章节末重复正文已经讲过的判断，是教科书章末总结腔 | 删掉整节。正文写清楚了，读者不需要复盘；下一章承接放在自然段落里 |
| 下一章解释... / 下一章将讨论... / 接下来一章会... | 章节末预告下一章是教科书衔接腔 | 删掉。下一章的标题和正文自己会说话 |

### 探讨口吻硬扫描

公众号长文尤其要避免"专家站上讲台"和"作者解释自己怎么写"。读者要看到问题、对象、动作和判断，不需要看到作者在正文里解释写作意图。

统一判据（2026-07-08 用户口径，比 `style_gate.py` 更严）：**凡是评述上文、防御定位、为下文摆姿态而不新增业务信息的句子，都算转场词，默认删除。**

| 禁用 / 慎用 | 问题 | 替换方向 |
|-------------|------|----------|
| 评述上文句（"数字站不住，担心是真的"） | 对上一段做评述，不新增业务信息 | 整句删除，直接进下一层事实 |
| 防御式定位句（"这些问题不拦着做试点"） | 替文章立场做防御，零信息 | 删除；真有边界就写清边界条件 |
| 贴标签框架句（"最先做的一批事和 AI 没有关系"） | 先贴标签再给内容 | 删标签直接写内容；事实句放内容后面收尾可以（"到这一步，还没有用到任何模型"） |
| 细节预告句（"跟着作业走一遍，卡的地方很具体""问题很实在"） | 为即将展开的细节摆姿态，零信息（2026-07-09 A/B 追认） | 删掉，细节直接上 |
| 背景很现实 / 更现实的是 | 抽象垫片，没有新增事实 | 直接写具体变化、动作或代价 |
| 我更关心的是 / 我不太想从...讲起 / 我想表达的是 / 本文想表达 | 作者意图说明，读者仍然没有得到信息 | 直接写你关心的对象、动作、边界或结论 |
| 这段话没有错，但... / 这也是问题所在 | 显性转场，像 AI 在解释自己换挡 | 删掉，直接写下一层业务事实 |
| 放到 X 场景里会更具体 / 从 X 视角来看 | 零信息桥接句，删掉不影响信息量 | 直接进入 X 场景里的具体事实 |
| X 卡住，往往卡在 Y / 关键在于 / 真正重要的是 | 专家教育式标题或判断，像在训话 | 标题改成具体问题、业务场景或动作链 |
| 临时创造 `X Agent` / `X 大脑` / `X 中台` | 未定义产品名会显得宽泛，还会把注意力从业务动作带走 | 用业务对象或系统动作描述，除非前文已定义且确有必要 |

处理方式：

- 能删就删。删掉后信息不减少的句子，默认就是 AI 味。
- 如果确实要表达取舍，直接写取舍后的动作和代价。
- 标题优先写"从一个门店转化低说起"这类具体入口，少写"AI 卡住，往往卡在..."这类专家式判断。
- 遇到 `X Agent` 这类命名，先问：这是已有产品、已有组织概念，还是临时造出来的比喻？后者默认改掉。

写作阶段的处理要求：

- 公众号长文：优先改成具体场景、真实观察、短句判断或读者处境。
- 商务工作：优先改成业务对象、动作、数据、边界、代价、责任人或待决策问题。
- 如果一个章节靠这些句式才能撑起节奏，说明本节缺少事实或判断，必须回到素材层补内容。

## 三、通用结构禁令

- 禁止连续使用 bullet point 罗列观点超过 3 条而不展开论述
- 禁止大段加粗（超过 2 行的加粗几乎肯定是过度结构化）
- 禁止教科书式开头（"在当今AI快速发展的时代"）
- 禁止空泛工具名/术语（不说"AI工具"、"某个模型"、"相关技术"，要说具体名字）
- 禁止编造事实、案例、数字——搜不到的信息诚实标注「待确认」或「暂缺」
- 禁止用“正确废话”做段落收束，例如“仍需持续优化”“后续继续推进”“有待进一步完善”。必须替换成下一步动作、责任边界、决策问题或待补数据。

## 四、商务工作专项禁用

商务工作中，以下"正确但空洞"的表达是 AI 味的重灾区：

| 禁用 | 替换建议 |
|------|----------|
| 取得显著进展 / 取得了阶段性成果 | 写具体完成了什么，还有什么没完成。例："核心目标达成，两个遗留问题待解" |
| 打下坚实基础 / 奠定了良好基础 | 写具体建成了什么能力，还差什么。例："完成了X系统搭建，Y能力尚未覆盖" |
| 系统性提升 / 全面加强 / 整体优化 | 写具体哪个维度提升了多少。例："当前优化解决了效率问题，体验侧还有两个关键缺口" |
| 持续推进相关能力建设 | 写具体推进什么、什么时间完成。例："近期完成A避免节奏影响；中期解决B防止规模瓶颈" |
| 进一步深化 / 不断完善 | 写具体深化什么、完善哪个方面 |
| 积极探索 / 大力推动 | 写具体探索了什么、推动了什么、结果如何 |
| 赋能 / 抓手 / 打造闭环 | 用具体动词描述具体动作 |

### 商务工作反 AI 核心原则

1. **禁止假完整**：不能写"一切搞定、路径清晰、无成本无约束"。真实的汇报一定有待共识点、分歧、阶段性局限
2. **禁止假正面**：亮点和不足必须同时存在，不回避问题
3. **禁止假精确**：没有数据时不要编数字，明确标注"待补数据"
4. **禁止万能结尾**：不用"继续努力、持续优化、不断提升"结尾，写具体的下一步动作和时间节点

### 保留真实元素

以下元素是商务工作的"活人感"来源，不要在润色中删除：

- 具体的时间节点（"7月份共识"、"10月中下旬上线"）
- 清晰的阶段划分（"从香港Launch到沙特Launch"）
- 资源/成本约束（"最小需要10人研发团队 + 50万美金数据采购"）
- 分歧或未决事项（"待共识点：是否采用端到端复用"）
- 短中长期路径差异
- 参考的具体公司和案例（Amazon、Uber、滴滴等）

## 五、公众号长文专项禁用

公众号长文采用商务工作风格基线（`voice/01-voice-dna.md`）+ 公众号专属增量（`content-article.md`）。详细风格规则见 `voice/01-voice-dna.md` 的禁忌词表和 `voice/04-values-and-antipatterns.md` 的反模式清单。以下是公众号语境下最高频踩坑的精简版：

| 禁用 | 替换建议 |
|------|----------|
| 下面我来给大家科普一下 | 知识直接用出来，不做科普预告 |
| 下面我来介绍 / 首先需要了解 | 直接写内容，不做预告 |
| 这给我们的启示是 | 删掉，直接写判断 |
| 我也面临这个问题 | 删掉。公众号长文禁止装朋友，用二人称"你"直击 |
| 聊着聊着想到了 / 让我想起 | 删掉。公众号长文不用情绪连接句 |
| 比如有一次...（编造案例） | 用真实咨询案例，或坦承"暂无公开数据" |
| 某种程度上 / 在一定意义上 / 本质上 | 说到底、其实、真相是 |
| 业内专家表示 / 相关研究显示 | 给出具体公司、具体人名、具体论文、具体年份 |
| 大概、应该、问题不大、差不多 | 具体数字、具体时间、具体百分比 |
| 取得显著进展、打下坚实基础 | 写具体完成了什么、还差什么 |

### 在场感的正确打开方式（无人称，2026-07-09 增）

第一人称禁令不变（"我也面临这个问题"仍禁，"我觉得/我认为"仍禁）。在场感来自细节颗粒度，不来自人称：

- 仍禁（第一人称口径）："我们做菜单项目时把录入时长估少了一半"——不因素材真实而放行
- 允许（无人称亲历级细节）："菜单项目立项时，单店录入时长被估少了一半，上线第三周才对出来"（可溯源到素材池条目）

亲历级细节只能来自素材池（`workflow-create.md` Step 1）："亲手做/亲眼看"条目可写过程和现场颗粒，"听说·二手"条目只写结果事实。素材池无料时显式降级（"这块还没跑过实际数据，只能先按 X 推测"），不硬造。定义与方法见 `human-rhythm.md`「人味来源」。

### 公众号长文反 AI 核心原则

1. **禁止故事开场/宏大叙事**：开头必须用"甩论断/二分拆解/反共识/抛冷事实/喊停套路"五种之一（见 `content-article.md` 第五章）
2. **禁止教科书科普**：知识直接用出来，不做"下面我来科普"的预告
3. **禁止装朋友**：作者是咨询顾问/教练口吻，不是朋友。禁止"我也面临这个问题"式的连接
4. **禁止升华癖**：结尾只允许"顺序清单/冷判断/类比定调"三种合法收法，禁止跑到文化/哲学/历史/人类命运（见 `content-article.md` 第八章）
5. **禁止模糊引用**：任何引用必须给具体公司、具体人名、具体论文、具体年份
6. **禁止假完整**：承认"现实解"、标注"待确认"、呈现张力，不假装方案完美无代价
7. **禁止碎句过度**：默认写自然段，只有章节首句总判断、章节尾句冷锤句、全文核心锤子句独立成段；不要把每个普通判断都打成单句段，由 `style_gate.py` 强制把关

完整反模式清单见 `voice/04-values-and-antipatterns.md`。

## 六、自检流程

每一节输出前先做一次小扫描；全文写作完成后，再执行完整扫描：

1. **章节输出前扫描**：检查本节是否命中二元对照、说教转场、空泛结尾；命中则先重写再输出
2. **全文搜索**：逐一检查上述禁用词表，发现则替换
3. **段落扫描**：检查是否有空泛段落（删掉后不影响信息量 = 应该删掉）
4. **数字检查**：所有数字/数据是否来自真实来源，是否标注了口径
5. **结尾检查**：结尾是否有具体的下一步动作（汇报）或具体的思考方向（公众号），空话收束必须删掉
6. **脚本检查**：已落盘正文必须跑 `style_gate.py`，失败则先返工
