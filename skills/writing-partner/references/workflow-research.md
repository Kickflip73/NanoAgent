# 调研报告流程

当商务工作分支的子类型选择为「调研报告」时加载本文件。

**与共创流程的区别**：共创流程中调研是写作的前置步骤；调研报告中，调研本身就是正文。调研的深度、结构和产出形态完全不同。

**前置要求**：执行本流程前，应已加载 `content-report.md`（会指引加载 `voice/` 风格源）。

**交互规则**：每一步完成后必须停下来等用户确认，绝对禁止自动进入下一步。

**产出规则**：
- 调研素材 → `99-materials/<季度>/<YYYYMMDD-项目名>/`（季度由 LLM 根据当前日期推算,通过 `resolve-path.sh --dir 99-materials/<季度>/<项目>` 获取路径 + Write/Edit 工具写入）
- 调研报告正文 → `03-outputs/reports/`（通过 `resolve-path.sh --dir 03-outputs/reports` 获取路径 + Write/Edit 工具写入）

---

## Step 1: 锁定调研范围

### 必须明确的信息

- **调研主题**：调研什么
- **核心问题**：这次调研要回答哪几个问题（列出 2-5 个）
- **调研边界**：包含什么、不包含什么（防止范围蔓延）
- **预期产出形态**：对比报告？选型建议？现状摸底？趋势分析？
- **汇报对象**：谁来读，谁来做决定
- **使用场景**：会前预读、正式汇报、决策支撑、沉淀文档

### AI 主动补充

如果用户给的信息足够明确，不追问，直接进入下一步。如果模糊，AI 主动提出建议：

> 基于你的描述，我理解调研范围是：[复述]
> 核心要回答的问题是：
> 1. ...
> 2. ...
> 3. ...
> 边界建议：包含 [X]，不包含 [Y]
> 你确认这个范围，还是需要调整？

---

## Step 2: 调研执行

**本步骤是调研报告的核心，不可跳过。**

### 搜索策略

1. 从 Step 1 的核心问题提取搜索关键词（中英文各 3-5 组）
2. 按三个范围展开调研：
   - **一手事实**：已有的内部数据、组织现状、历史决策、已知约束
   - **外部资料**：行业文章、研究报告、公司案例、专家讨论、开源项目
   - **反向证据**：失败案例、反对观点、落地阻力、已知局限
3. 学术类主题必查 arxiv

### last30days 最近 30 天社区信号（按需增强）

如果调研范围涉及 AI 演进、行业趋势、工具/技术选型、开源项目、竞品动作、社区反馈，或用户明确要求“看最近大家怎么说”，加载 `references/research-last30days.md` 并叠加 `last30days` 调研。

执行要求：
- 先检测 `last30days` 是否已安装；未安装时按 `research-last30days.md` 的规则安装或征求用户确认
- 从 Step 1 的核心问题中提炼 1-3 个 `last30days` 主题
- `last30days` 原始结果可以保留英文，但调研发现、判断、边界、写入素材必须用中文表达
- 按 `research-last30days.md` 的覆盖与降级门禁执行；关键源不可访问时先诊断和排障，不能静默略过
- 把结果归类为“社区信号”或“反向证据”，不要把热度直接写成事实结论
- 将 raw 结果路径和关键来源 URL 记录进 `99-materials/<季度>/<项目>/调研素材.md`
- 完成后停下让用户确认，再进入 Step 3

### 子 Agent 并行搜索

对于复杂调研任务，使用子 Agent 并行搜索提高效率：

- **子 Agent 1**：搜索主题相关的核心资料（行业报告、权威分析、官方文档）
- **子 Agent 2**：搜索竞品/案例/对标信息
- **子 Agent 3**（复杂主题才需要）：补充信息（社区讨论、反向证据、学术论文）

子 Agent 工具指引：
- WebSearch 用于发现信息来源和线索
- WebFetch 用于从具体 URL 定向提取内容
- 先用 WebSearch 发现，找到 URL 后用 WebFetch 深入提取
- 一手来源优于二手来源：官方博客 > 权威媒体原创 > 转载/聚合

### 素材记录格式

每条素材记录：
- 标题、作者/机构、时间、URL
- 核心观点（2-3 句话）
- 可支撑的判断（这条素材能支撑 Step 1 中的哪个核心问题）
- 不可直接套用的边界（这条素材的局限性）

### 素材保存

调研过程中，素材分批保存到 `99-materials/<季度>/<YYYYMMDD-项目名>/调研素材.md`(季度由 LLM 推算)：

```bash
# 获取素材文件路径(以 2026/05 在做"商分知识库"项目为例)
bash skills/writing-partner/scripts/resolve-path.sh --dir 99-materials/2026Q2/20260505-商分知识库 "调研素材.md"
# 输出绝对路径后，用 Write 工具首次创建，用 Edit 工具追加
```

素材文件结构：
```markdown
# {主题} 调研素材

## 调研范围
[从 Step 1 复制]

## 素材清单

### 素材 1: {标题}
- 来源：{作者/机构}, {时间}, {URL}
- 核心观点：...
- 可支撑判断：...
- 边界：...
```

### 调研完成检查

- [ ] 每个核心问题都有至少 2-3 条相关素材
- [ ] 每条素材有完整引用信息（标题、作者、时间、URL）
- [ ] 包含反向证据（不只是支持性材料）
- [ ] 事实与推演已区分
- [ ] 素材文件已通过 resolve-path.sh 校验路径后写入 `99-materials/<季度>/<项目>/`

---

## Step 3: 发现提炼与判断形成

### 从素材到判断

基于 Step 2 的素材，提炼关键发现。每个发现用以下结构呈现：

```
### 判断 N：{一句话结论}

**依据：**
- 事实：{调研中发现的具体事实/数据}
- 对标：{竞品/行业怎么做的}
- 数据：{量化支撑——成本、周期、覆盖率等}

**边界：** {这个判断在什么条件下不成立，或有什么局限}

**置信度：** 已验证 / 初步判断 / 待确认
```

### 对比分析（如适用）

如果调研涉及多方案/多产品/多技术的对比，**强制使用表格**：

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| ... | ... | ... | ... |

表格后附文字说明：推荐哪个、为什么、在什么条件下推荐会变。

### 产出

本步骤产出「关键判断清单」，每条判断带一句话依据。这个清单将成为 Step 4 调研报告的「调研结论摘要」章节。

向用户展示判断清单，确认后进入 Step 3.5。

---

## Step 3.5: 报告大纲蓝图确认

基于 Step 3 的判断清单，生成报告大纲蓝图，向用户确认后再进入成稿。**禁止跳过本步骤直接开写。**

### 大纲蓝图包含

1. **报告标题**（2-3 个备选）
2. **章节结构**：每章标题 + 一句话说明写什么 + 预估篇幅
3. **判断分配**：哪些判断在哪章展开，哪些精简处理
4. **附加章节**：是否需要对比表格、场景矩阵、附录等
5. **总预估篇幅**

### 用户决策

- 章节顺序是否合理
- 哪些判断需要重点展开，哪些可以精简
- 是否需要增删章节
- 篇幅预期

确认后进入 Step 4。

---

## Step 4: 调研报告成稿

**基于 Step 3.5 确认的大纲蓝图，逐节写作。**

### 报告结构

```markdown
# {调研主题} 调研报告

## 一、调研结论摘要

[一页纸能看完。核心判断列表，每条带一句话依据。读者看完这一节就知道所有关键结论。]

判断1：{结论}。依据：{一句话}。
判断2：{结论}。依据：{一句话}。
判断3：{结论}。依据：{一句话}。

## 二、调研范围与方法

- 调研主题：...
- 核心问题：...
- 调研边界：包含 X，不包含 Y
- 信息来源：{来源类型和数量}
- 局限性：{本次调研未覆盖的、数据不足的}

## 三、关键发现

[每个发现用「判断→依据→边界」结构展开。这是报告的主体部分。]

### 发现 1：{判断}

依据：
1. {事实}
2. {对标}
3. {数据}

边界：{局限}

### 发现 2：...

## 四、对比分析

[多方案/多产品时强制表格。单一主题调研可省略本章。]

## 五、建议与下一步

[基于调研发现给出明确建议。每条建议对应一个判断。]
```

### 写作风格

- 加载 `content-report.md` + `voice/` 系列的所有风格规则
- 结论先行：每章、每段先给判断，再展开
- 编号递进：判断1 / 判断2 / 判断3
- 数字驱动：具体数据替代模糊表达
- 表格对比：多方案对比强制用表格
- 克制措辞：无"我觉得"、无模糊程度词

### 逐节写入（强制）

每个章节独立完成、独立确认、独立写入。禁止累积多章再统一写入。

流程：
1. 内部完成一个章节的候选稿，并先做去 AI 味返工、`style_gate.py` 或等效片段扫描、自检和必要 peer review
2. 只把通过质量门禁后的章节候选稿展示给用户确认
3. 等待用户确认（"确认/OK/继续"或修改意见）
4. 用户确认后，**立即写入文件**（通过 `resolve-path.sh` 获取路径 + Write/Edit 工具）
5. 写入后再跑一次 `style_gate.py` 做落盘回归检查；如失败，先修复并重新给用户确认，不继续下一章
6. 写入和回归检查都通过后，再开始下一个章节

违规判定：如果一次输出中包含超过一个章节的完整内容，即为违规。

### Peer review 节奏（强制）

调研报告成稿阶段是关键实现步骤，必须走 Agent Bridge peer review，不能只靠 style_gate。

触发节奏（选其一）：

- **逐章 review**：单章长度 ≥ 2500 字、或包含主线判断/关键论证时，章节落盘前找 peer agent review，确认脉络成立、事实可核、AI 味无残留，再请用户确认
- **批量 review**：所有章节长度都 < 2500 字时，可每 2-3 章成稿后做一次合并 review，但不晚于第 3 章累积时点

Peer review 内容必须包含 5 条独立判断要求：脉络成立性、事实硬伤、漏掉的反驳点、AI 味残留、被忽略的视角。Codex 给的 P0/P1 issue 必须在用户确认前修掉；P2 issue 可挂到 Self-Iteration Notes 里下一次处理。

跨 agent 协作走 Agent Bridge MCP（`mcp__plugin_agent-bridge_agent-bridge__codex_send` 走 review mode），禁止 `claude -p / claude --print`。

```bash
# 获取报告文件路径(调研报告归 reports)
bash skills/writing-partner/scripts/resolve-path.sh --dir 03-outputs/reports "YYYYMMDD-{主题}-调研报告.md"
# 输出: /absolute/path/to/03-outputs/reports/YYYYMMDD-{主题}-调研报告.md

# 首次创建：用 Write 工具写入上述路径
# 后续追加：用 Edit 工具在文件末尾追加（old_string 匹配文件最后一段，new_string 为原内容+新章节）
# 或用 resolve-path.sh --append 确认文件存在后，用 Write 工具重写完整内容
```

---

## Step 5: 质检收尾

### 判断链完整性检查

- [ ] 每个判断都有明确依据（事实/对标/数据），不是"我觉得"
- [ ] 每个判断都标注了边界/局限
- [ ] 结论摘要与正文判断一致，没有遗漏或矛盾
- [ ] 对比分析（如有）用了表格，维度完整
- [ ] 事实/初步判断/待确认已区分

### 反对者压力测试

> 现在扮演读者中的质疑者，找出这份调研报告的所有弱点。
> 1. 哪些判断的证据最薄弱？
> 2. 有没有明显的替代解释没有被讨论？
> 3. 数据来源是否经得起追问？
> 4. 调研范围是否有盲区？

### 成稿收尾

1. 检查标题是否匹配调研核心发现
2. 如标题已调整，更新文件名
3. 检查所有 `待补信息` / `待确认` 标签，提醒用户处理
4. 最后一轮通读：压缩重复表达，检查小标题质量
5. 用 resolve-path.sh + Write 工具写入最终版本
