# 共创流程（从零开始）

当工作模式选择为「共创」时加载本文件。适用于从零开始的写作，无论内容类型是公众号长文还是商务工作。

**前置要求**：执行本流程前，应已加载对应的内容类型风格规则（`content-article.md` 或 `content-report.md`）。

**交互规则**：每一步完成后必须停下来等用户确认，绝对禁止自动进入下一步。

---

## Step 1: 锁定目标

### 通用必答

无论内容类型是什么，都需要明确以下信息。如果用户已经给得足够明确，不需要追问，直接进入下一步：

- 主题是什么
- 当前处于什么阶段（如适用）
- 最想讲清的核心判断是什么

### 公众号长文补充

- **目标读者**：谁来读？他们知道什么？关心什么？在什么场景下读？
- **核心问题**：这篇文章要解决什么问题？解答什么疑惑？满足什么需求？
- **核心判断（灵魂）**：用一句话说出你的核心观点。如果没有独特的认知和判断，AI 只会生成空洞内容。
- **人味素材挖掘（共创强制；三问是唯一硬步骤，预检索是可选增强）**：
  1. （可选增强，仅当前仓库存在 `scripts/wiki-index.py` 等个人知识库索引时）先用 `wiki-query` 检索用户 worklog / wiki 里与主题相关的一手记录，列给用户勾选："这里面哪些是你亲手做的？"本 skill 会被复制到没有 wiki 索引的环境使用——脚本不存在或查询失败时**静默跳过本步，直接三问**，不把预检索当依赖。
  2. 三问（硬步骤，可一次性回答）：
     - 这件事里，你亲手做过 / 亲眼看到的部分是什么？
     - 哪里让你意外、不舒服、或者想吐槽？
     - 哪个结论你自己其实也没把握？
  3. 勾选结果和三问答案写入素材池（见下方"素材池"）。写作阶段只能消耗池内素材，禁止编造亲历细节；用户暂时答不出也可以先推进，正文按"无料降级"写法处理（见 `human-rhythm.md` 人味来源）。

### 商务工作补充

- **汇报对象**：谁来读，谁来听，谁来做决定
- **使用场景**：会前预读、正式汇报、答辩、同步进展、沉淀文档
- **目标**：希望读完之后形成什么结论、共识或动作
- **核心判断**：这次最想传递的判断是什么
- **最大约束/分歧/风险**：当前面临的关键约束、内部分歧或风险是什么
- **亲历卡点**：这件事推进中你亲历的卡点或返工是什么？（答案入素材池，用于在场感）
- **材料子类型**：周报/述职/讲稿/PPT/阶段汇报/复盘/规划/OP（调研报告和技术方案走专属流程）

### 素材池（一手素材.md）

素材挖掘的产出经 `resolve-path.sh` 落盘为 `99-materials/<季度>/<YYYYMMDD-项目名>/一手素材.md`，逐条记录：

```markdown
- [S1]（亲手做｜2026-07-08 口述）"当时菜单录入的时长被估少了一半，上线第三周才对出来"
- [S2]（亲眼看｜worklog 2026-06-30）"运营在两个系统间来回切，一单菜单要开五个页面"
- [S3]（听说·二手｜wiki entities/xxx）"客服侧也有类似的草稿需求"
- [没把握] "AI 那部分的收益到底怎么和流程改造分账，还没有共识算法"
```

规则：

- **记用户原话，禁 AI 转述改写**——用户语言的指纹（用词、句长、口径）是唯一不可伪造的人味来源
- 亲历关系必标（`亲手做 / 亲眼看 / 听说·二手`）：前两类可写过程和现场颗粒，二手只写结果事实
- "没把握"条目单独标注，是正文"真实不确定"的唯一合法来源
- 共创/打磨过程中用户口述的任何经历性陈述，**当场追加进素材池**，再进入正文
- 正文消耗素材时标注来源编号（如 S1）便于质检溯源；编号只允许出现在内部草稿和质检版本，给用户确认的候选稿、预览版、发布版必须清零

## Step 2: 调研

**本步骤默认执行，不是可选项。** 只有明确符合「可豁免」条件时才能跳过。

### AI 自查清单（询问用户前必做）

在与用户对话之前，AI 先对照以下信号自查。命中任意一条即判定为「必须调研」：

- [ ] 用户在 Step 1 描述中出现"选型 / 对比 / 方案 / 落地 / 调研 / 研究 / 技术方案 / 横向对比"等关键词
- [ ] 用户给出了多个候选技术/产品/工具，需要做选择
- [ ] 用户表达了不确定（"不知道哪个好 / 想搞清楚 / 帮我看看"）
- [ ] 主题涉及行业趋势、AI 演进、组织变革、外部对标、竞品动作
- [ ] 用户明确说"先调研"
- [ ] 涉及不在用户知识范围内的新领域（用户说"我不太了解 X"）
- [ ] 用户明确提到"落地/可执行/技术方案/实施"但缺少事实基础

### 询问方式（命中 vs 未命中）

**命中「必须调研」信号时**，AI 主动告知并给理由，不再提供跳过选项：

> 本次任务属于必须调研的场景（理由：[具体触发了哪条，结合用户原话说明]）。
> 这类场景不能直接进入蓝图或正文。下一步先确认调研范围，然后开始调研。
> 如果你认为已有完整事实基础，请先补充事实来源、数据边界和已验证结论；AI 重新判断是否满足「可豁免」条件。

**未命中任何信号时**，才可以问：

> 本次任务默认可以豁免调研（理由：[为什么判断不需要]）。
> 是否确认跳过调研，直接进入写作？

### 可豁免场景（严格限定）

只有同时满足以下条件才可豁免：
- 周报、阶段同步、简单复盘、日常纪要
- 用户已提供完整事实、数据、结论
- 重点是表达和结构，不是外部证据补强
- **没有任何一条「必须调研」信号被触发**

### 必须先调研的场景

- 上述 AI 自查清单中任一信号命中
- 主题涉及行业趋势、AI 演进、组织变革、外部对标、竞品动作
- 当前观点需要外部案例、公开数据、最新变化来支撑
- 结论本身争议较大，需要先看反例和约束条件
- **技术选型 / 工具对比 / 方案设计 / 架构决策**（即使用户没明确说调研）

### 调研执行（按内容类型分支）

#### 公众号长文调研

目标：找到有影响力的近期文章/资料，为蓝图生成和写作提供素材支撑。

流程：
1. 从 Step 1 提取 3-5 组搜索关键词
2. GitHub 搜索（重点）：高星项目、README/Wiki、Issues/Discussions
3. 英文资料搜索（主力）：权威媒体、学术论文、报告、知名博客
4. 中文资料搜索（辅助）：国内有影响力的深度分析
5. 整理每个来源：标题、作者/机构、时间、URL、核心观点

要求：
- 英文资料占比 70%+，近两年，3-5 组不同角度搜索
- 来源优先级：GitHub 高星项目 > 权威英文媒体 > 中文深度分析
- 学术类主题必查 arxiv：`curl -s "https://export.arxiv.org/api/query?search_query=all:关键词1+AND+all:关键词2&max_results=10"`，返回 XML 格式，含标题、作者、摘要、PDF 链接
- 总结 3-5 条最有价值的发现

#### 商务工作调研

目标：让后续判断更稳，重点回答四个问题：
1. 外部环境到底发生了什么变化
2. 有哪些可类比案例，哪些不能直接类比
3. 这些变化对当前汇报主题意味着什么
4. 哪些是事实，哪些只是推演

范围：
- 内部事实：已有组织痛点、历史调整、当前协作链路、已知约束
- 外部资料：行业文章、研究报告、公司案例、专家讨论
- 反向证据：失败案例、反对观点、落地阻力
- 学术类主题：arxiv 论文（同公众号调研的 arxiv 查询方式）

每条素材记录：标题、作者/机构、时间、URL、核心观点、可支撑的判断、不可直接套用的边界

#### 可选：last30days 最近 30 天社区信号

当主题涉及 AI 演进、行业趋势、工具/技术选型、竞品动作、开源项目、社区反馈，或用户明确要求“看最近大家怎么说”时，加载 `references/research-last30days.md`，在常规调研之外叠加 `last30days`。

执行要求：
- 先检测 `last30days` 是否已安装；未安装时按 `research-last30days.md` 的规则安装或征求用户确认
- 从当前写作目标中提炼 1-3 个具体调研主题，不要把整篇写作需求原样丢给 `last30days`
- `last30days` 的原始结果可以保留英文，但展示给用户和写入素材的发现、判断、边界必须用中文
- 按 `research-last30days.md` 的覆盖与降级门禁执行；X、Reddit 等关键源不可访问时，先诊断和排障，不能静默略过
- 结果只作为“社区信号/反向证据/最近实践案例”，不能替代官方文档、GitHub、内部事实或论文
- 调研完成后必须停下让用户确认，不得自动进入蓝图或正文

#### 可选：横纵分析法（适合产品/技术/竞品分析类深度调研）

当调研主题涉及产品、技术选型、竞品对标、系统演进等方向，且需要系统性的深度研究时，可在本流程内叠加横纵分析法。

横纵分析法提供双轴视角：**纵轴**追踪从诞生到当下的完整发展脉络（叙事驱动），**横轴**在当下时间截面做系统性竞品对比，最后**交叉洞察**产出新判断。

使用方式：
- **完整执行**：当调研本身就是汇报的主要内容（如技术调研报告、竞品分析报告），在调研素材中增加「纵轴演进 / 横轴对比 / 交叉洞察」三节，先完成研究判断，再进入写作
- **部分借用**：当只需要横纵视角辅助思考，可以只用纵轴（讲系统演进）、只用横轴（讲竞品对比）、或两者结合，不必走完整流程

执行要求：纵轴必须给出时间线和关键拐点；横轴必须给出对比对象、维度和差异表；交叉洞察必须落到当前写作主张，不能只停留在资料罗列。

### 子 Agent 并行搜索策略

对于复杂调研任务，可使用子 Agent 并行搜索提高效率：

- **子 Agent 1**：搜索主题相关的核心资料（行业报告、权威分析）
- **子 Agent 2**：搜索竞品/案例/对标信息
- **子 Agent 3**（复杂主题才需要）：补充信息（社区讨论、反向证据、学术论文）

子 Agent 工具指引：
- WebSearch 用于发现信息来源和线索
- WebFetch 用于从具体 URL 定向提取内容
- 先用 WebSearch 发现，找到 URL 后用 WebFetch 深入提取
- 一手来源优于二手来源：官方博客 > 权威媒体原创 > 转载/聚合

### 调研输出

保存到：`99-materials/<季度>/<YYYYMMDD-项目名>/调研素材.md`

季度由 LLM 根据当前日期推算（1-3 月→Q1, 4-6 月→Q2, 7-9 月→Q3, 10-12 月→Q4）。

**必须**通过 `resolve-path.sh` 校验路径后用 Write/Edit 工具写入：

```bash
# 获取素材文件路径(以 2026/05 在做"商分知识库"项目为例)
bash skills/writing-partner/scripts/resolve-path.sh --dir 99-materials/2026Q2/20260505-商分知识库 "调研素材.md"
# 输出绝对路径后，用 Write 工具首次创建，用 Edit 工具追加
```

调研完成后的检查清单：
- [ ] 每个来源有完整引用信息（标题、作者、时间、URL）
- [ ] 总结了 3-5 条最有价值的发现
- [ ] 文件通过 resolve-path.sh 校验路径后写入 `99-materials/<季度>/<项目>/`
- [ ] （商务工作）事实/判断/待确认已区分
- [ ] （商务工作）调研对当前决策的意义已写明

### 踩坑提醒（事后反思钩子）

如果 Step 2 因满足「可豁免」条件而跳过调研，后续流程中一旦出现以下信号，**立即回到 Step 2 补做调研**，不要硬撑：

- 需要让用户在多个候选项之间做选择（"你想用 A 还是 B"）
- AI 开始使用"建议"、"推测"、"一般来说"等没有事实支撑的表达
- 用户问"这个对吗 / 你确定吗"时，AI 无法给出一手来源
- Step 3 蓝图里出现大量 `待确认` 标签

出现以上任一信号说明当时该调研而没调研，必须补做，**不要用"用户凭感觉拍板"替代"基于事实判断"**。

## Step 3: 生成蓝图 / 结构化草稿

### 公众号长文

基于 Step 1 的创作目标和 Step 2 的调研素材，生成蓝图：

1. **5 个标题备选**（风格多样）
2. **核心价值主张**（一句话版 + 段落版）
3. **3-5 个分论点方向**，每个包括：
   - 论点概述
   - 可能的案例/数据方向
   - 潜在风险
4. **整体结构建议**（线性推进？并列？问题-解决？）

用户决策：选哪个标题？哪些分论点？重点放在哪里？

### 商务工作

基于已有信息，构建判断链：

1. **当前阶段定位**
2. **核心判断**（1-3 条）
3. **每条判断的决策依据**
4. **关键约束和取舍**
5. **下一步动作和时间节点**

如果信息不全，先出结构化草稿，标注待补项（`待补信息：`、`待确认数据：`、`待确认判断：`、`待补案例：`）。禁止为了完整而编造。

## Step 4: 细化大纲

### 公众号长文

基于选定的蓝图，展开段落级大纲：

```markdown
# [文章标题]

## 开头（~300字）
- 核心任务：让读者在第一段看到对象、动作和处境
- 内容设计：
  - 开场：___（默认：具体业务场景进入；备选：甩论断/二分拆解/反共识/抛冷事实/喊停套路，见 content-article.md 第五章）
  - 问题：___（文章要解决什么问题？）
  - 核心判断落位：___

## 第一部分：XXX（~800字）
- 核心观点：___
- 支撑材料：
  - 案例：___
  - 素材池条目：___（消耗哪几条一手素材，标编号；本部分无亲历素材则写"纯推理"）
  - 数据：___
- 预计字数：___
- 衔接方式：硬切（默认；章节之间不写过渡句）

## 第二部分：...
...

## 结尾（~300字）
- 总结：回扣核心观点
- 公众号长文：用顺序清单、冷判断回扣或类比定调三种合法收法之一收束，避免文化升华
- 行动/思考：给读者具体方向

---
**预计总字数**：___ 字
```

检查：
- [ ] 结构是否完整覆盖想表达的内容？
- [ ] 各部分逻辑顺序是否合理？
- [ ] 每个部分是否有足够的素材支撑？
- [ ] 素材池条目数与章节数匹配吗？亲历素材撑不满全部章节时，当场标注哪几章走纯推理，或回 Step 1 再挖；单个案例最多贯穿两章
- [ ] 开头提的问题是否在结尾被回答？

### 商务工作

按材料子类型微调结构，确保：
- [ ] 核心判断足够前置
- [ ] 约束/分歧/风险已写入
- [ ] 不存在只有标题没有论证的段落
- [ ] 下一步动作和待决策点明确
- [ ] 未来目标有时间节点和阶段划分

## Step 5: 分段写作

一节一节推进，先完成再完善。**每个章节展示给用户确认前，必须先完成去 AI 味、style gate 或等效扫描、自检和必要 review；用户确认后必须立即写入文件，不允许累积多章再统一写入。**

### 风格前置检查（公众号长文 / 商务工作）

商务工作和公众号长文都以 `voice/01-voice-dna.md` 为风格基线。本步骤开始前：

- 商务工作：确认已加载 `references/style-report-voice.md` 及 `voice/01-voice-dna.md`
- 公众号长文：确认已加载 `voice/01-voice-dna.md` + `references/content-article.md`（公众号专属增量）+ `references/human-rhythm.md`

每节输出前，对照以下清单自查：

- [ ] 结论先行了吗？先给判断，再展开依据
- [ ] 段落以自然段为主吗？仅章节首句总判断、章节尾句冷锤句、全文核心锤子句独立成段？
- [ ] 判断组织自然吗？需要密集判断时才用"判断1/判断2/判断3"，普通段落优先写自然判断句
- [ ] 数字具体了吗？无"大概""应该""问题不大"
- [ ] 禁忌词清零了吗？无"我觉得""我认为"
- [ ] 二元对照清零了吗？用户可见正文默认不出现这类 A/B 纠偏句
- [ ] 素材池有料时，本节消耗了亲历级细节吗？（无人称颗粒；无料已按降级写法处理——见 `human-rhythm.md` 人味来源）
- [ ] 公众号长文额外：章节用硬编号？开头用默认打法（具体场景进入）或五种备选之一？涉及"值不值得做"用 ROI 账本？结尾用"顺序清单/冷判断/类比定调"？

### 每节去 AI 味闸门

每节输出给用户之前，必须先在内部完成一轮"初稿 -> 扫描 -> 重写 -> 再输出"。公众号长文加载 `article-de-ai-workflow.md` 执行七步返工（含"作者在场证据"步：素材池消耗与降级判断）；用户只看到重写后的章节正文，不要把内部检查表暴露出来，除非用户要求看修改说明。

用户确认前的版本必须是当前 agent 能交付的候选终稿，不是等待用户确认后再清理 AI 味的半成品。用户确认只用于内容判断、观点取舍、事实补充、是否写入文件和是否进入下一节；不能把去 AI 味、style gate、自检这类基础质量工作放到用户确认之后。

内部扫描时重点查这些高频 AI 腔：

- A/B 纠偏句、递进升格句：默认删除或改成直接判断；确需保留时必须在质检报告说明原因
- "有一个很关键"、"关键在于"、"真正重要的是"：改成具体判断 + 影响，不要先摆姿态
- "这里面每一步都不神秘"、"并不复杂"、"说穿了"：改成具体流程、真实难点或可执行动作
- "值得注意的是"、"更重要的是"、"本质上"、"这意味着"：能删就删，不能删就换成事实、因果或边界句
- 连续三段都是抽象判断：补事实、业务对象、动作、约束、例子或待决策问题；补不出来就压缩

本节通过闸门后才能输出。没有通过时，先重写，不要把"后续再润色"留到完稿阶段。

未落盘片段也要做等效质量门禁；可以写入 `tmp/writing-partner-stylecheck/` 等临时文件运行 `style_gate.py`，或按脚本规则做同等扫描。脚本失败时先返工，再展示给用户确认。

落盘后还要再次运行：

```bash
python3 skills/writing-partner/scripts/style_gate.py <markdown-file>
```

脚本失败时，先按命中行返工，并把修复后的候选稿重新给用户确认；不能继续下一节。

### 流程

1. 选择一个章节（通常从开头或第一部分开始）
2. 用户提供：想法、素材、草稿（哪怕很粗糙）；口述中的经历性陈述当场追加进素材池
3. AI 内部起草并执行"每节去 AI 味闸门"
4. AI 在内部完成快速检查（见下方四维检查）和 `style_gate.py` / 等效片段扫描
5. AI 只输出通过质量门禁后的完整章节候选稿（按已加载的风格规则执行）
6. 章节确认时，顺带邀请用户"用你自己的话说一遍这章核心判断"（用户可跳过）；用户原话可直接替换正文对应判断句，并追加进素材池
7. 用户满意后，**立即写入文件**（通过 `resolve-path.sh` 获取路径 + Write/Edit 工具）
8. 落盘后再次运行 `style_gate.py`；失败则修复当前章节并重新给用户确认
9. 写入和门禁都通过后再进入下一节

### 每节四维快速检查

**1. 观点概括检查**
- AI 用一句话总结本节核心观点
- 对比：是否与创作意图一致？不一致则调整

**2. 案例/证据具体度检查**
- 所有案例是否具体（有时间、地点、人物、动作、结果）？
- 数据是否可靠？来源是否标注？

**3. 逻辑衔接检查**
- 本节与上节的逻辑衔接是否成立？（靠判断和事实的顺序衔接，不靠桥接句）
- 禁止用"说完了 X，再看 Y""在此基础上"这类过渡句缝合；章节之间硬切
- 如果读起来跳跃，说明论证链缺环，回素材层补内容，不是补过渡句

**4. 表达活人感检查**
- 是否出现 A/B 纠偏句、"有一个很关键""每一步都不神秘"这类说教句式？
- 每个抽象判断后是否接了事实、动作、边界、代价或待决策问题？
- 本节是否至少有一个具体业务对象、具体动作、具体数据、具体例子或具体待确认项？
- 如果像 AI 在讲道理，先按 `human-rhythm.md` 的返工动作重写，再输出给用户。

### 效率提示

- 不要空手来——给 AI 一些输入（想法、素材、粗糙草稿）
- 接受不完美——初稿不需要完美，先完成全文
- 保持连续——提供上节最后几句话帮助 AI 衔接
- 信息不全时不要停——先出结构化草稿，标注待补项

## Step 7: 精修 / 反驳 / 升维

### 公众号长文

#### Round 1: 结构精修
- [ ] 每节核心观点是否清晰？
- [ ] 论据是否充分？
- [ ] 逻辑是否自洽？
- [ ] 节奏是否合适（拖沓或过简的部分）？
- [ ] 核心观点是否得到充分展开？

#### Round 2: 表达精修
- [ ] 语言是否精准？
- [ ] 是否有冗余（删掉后信息量不减）？
- [ ] 比喻是否恰当？
- [ ] 句式是否有变化？
- [ ] 有没有值得截图或转发的判断句？

#### Round 3: 平台适配
- [ ] 开头是否在 3 秒内抓住注意力？
- [ ] 段落以自然段为主，仅章节首尾判断和核心锤子句独立成段？（避免每个判断都打成单句段）
- [ ] 每 300-500 字是否有一个留住点（断言、反例、账本、对比）？
- [ ] 公众号长文结尾是否落在"顺序清单 / 冷判断回扣 / 类比定调"三种合法收法之一？

#### 反对者压力测试

> 现在扮演反对者，找出这篇文章的所有弱点。
> 列出：1) 3-5 个最强有力的反驳点 2) 每个反驳点的严重程度（1-5分） 3) 文章当前是否有回应？ 4) 如何加强回应？

用户决策：哪些反驳是真实存在的？哪些需要在文中回应？

#### 收尾方式校准

公众号长文默认不做文化/哲学/历史升维。检查结尾是否落在顺序清单、冷判断回扣或类比定调上，避免空泛呼吁。详见 `content-article.md` 第八章"反升华结尾"。

### 商务工作

#### 判断链完整性检查
- [ ] 阶段→判断→依据→约束取舍→动作，每个环节是否都写清了？
- [ ] 亮点和不足是否双面呈现？
- [ ] 待共识点和未决事项是否标注？

#### 反对者压力测试

> 现在扮演读者中的质疑者，找出这份材料的所有弱点。
> 列出：1) 3-5 个最可能被挑战的判断 2) 每个判断的证据是否充分？ 3) 有没有明显的替代方案没有被讨论？ 4) 数据是否经得起追问？

#### 数据补充
- 哪些论述最需要数据支撑？
- 标记优先级（哪个最影响说服力？）
- 建议搜索什么类型的数据？

#### 逻辑强化
- A 到 B 的推导是否成立？
- 是否有跳跃需要补中间步骤？
- 是否有隐含假设需要显式化？

用户决策：选择性采纳——不是所有建议都需要执行。
