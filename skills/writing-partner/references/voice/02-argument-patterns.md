# 论证模式与决策启发式

组织论证、搭判断链时加载。定义商务工作中"怎么组织观点、怎么展开论证"的范式。

---

## 核心论证结构：先 Fact 再 Judgment

**这是最重要的论证范式。**

事实铺陈 → 竞品对标 → 数据支撑 → 再给判断 → 最后是进展和下一步。不能跳过前面直接给结论。

示例：
> **课题：自建地图 vs 外采地图**
>
> 事实：某地图供应商虽然好用，但会随客户业务规模增长后，收费模式从按请求量转为按订单量计费。
>
> 竞品对标：参考 Uber/Grab 的做法，均在业务达到一定规模后启动了地图自建。
>
> 数据支撑：使用自建地图的拐点是10~15万单（外采成本年均1000万人民币，大致可以覆盖10人地图团队+数据采买）。
>
> 判断：长期必须建自建替换能力，保留议价空间。
>
> 进展与下一步：已完成基础能力搭建，下一阶段重点是覆盖率提升和成本优化。

## 7 条决策启发式

### 1. 看"该谁干"，不看"能不能干"

专人做专事。基础设施归基础设施团队、平台归平台团队、业务归业务团队。

示例：PaaS 上云由基础研发团队统一做，业务研发专注业务迭代。

### 2. 按"核心价值+专业度+迭代热度+关联度"四维度评估复用

按领域拆解复用边界。

示例：商品（专业度高、迭代慢）全迁，营销（迭代热、多样化）新建，订单（底层稳定、上层差异大）底座复用。

### 3. 先讲 Fact 再给 Judgment（核心论证范式）

事实铺陈 → 竞品对标 → 数据支撑 → 再给判断 → 最后是进展和下一步。不能跳过前面直接给结论。

### 4. 预判分歧，主动标注"待决策"

在汇报/方案中主动暴露"当前共识/当前分歧/待决策"，不要藏着掖着等别人追问。对未决事项给出 A/B/C 多套方案 + 推荐理由。

示例：客服系统的 A 外采 / B 公司承接 / C 联合共建三方案对比。

### 5. 对"看起来先进但方向错误"的方案敢说不

先穿透到问题本质——这个方案解决的是真问题吗？表层繁忙不等于方向正确。如果问题被定义错了，叠再多复杂度也是白搭。

示例：Agent Team 解决多仓 AI Coding——判断"问题被错误定义了"，否决。真正难点在跨仓架构和业务关系认知。

### 6. 用"亮点/不足"做对称性自评

每个大模块都给出亮点和不足。不足要具体（"投入不足""启动过晚""过于激进"），不能只是客套话。承认"一线研发比较疲惫"这类对人的观察。

### 7. 长期正确优先于短期舒适

"做难而正确的事情，用短痛避免长痛。"

示例：某研发中心搬迁、大数据出海全量迁移、Region 改造都是短期痛、长期收益的决策。

## 论证中的方案呈现范式

### 多方案对比（必须用表格）

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 成本 | ... | ... | ... |
| 周期 | ... | ... | ... |
| 风险 | ... | ... | ... |
| 推荐理由 | ... | ... | ... |

### 分层判断（编号递进）

> 判断1：相对于方案A，方案B更合适。原因是……
>
> 判断2：在方案B的基础上，需要额外补充……原因是……
>
> 判断3：长期终态是……当前路径先……

### 待决策项呈现

> **待决策**：关于X的路径选择
>
> - 当前共识：……
> - 当前分歧：……
> - 方案A：……（推荐，理由是……）
> - 方案B：……（备选，代价是……）
