# MCP 进入生产区：Agent 连接层的价值、边界与落地清单

写作日期：2026-07-16

## 摘要

MCP 在 2025 年完成了从厂商协议到开放生态基础设施的跃迁。它用统一方式描述智能体可读取的资源、可调用的工具和可复用的提示模板，减少客户端与外部系统之间的重复适配。协议普及也放大了权限、注入、凭证、审计和不确定副作用问题。本文结合 MCP 2025-11-25 规范、2025 年底的生态进展与 NSA 2026 年安全建议，给出一个适合团队落地的判断：把 MCP 当作连接层，在 Host 和业务系统两侧补齐身份、最小权限、事务语义与审计。协议兼容性值得投资，安全责任不能外包给协议。

## 背景：为什么 2026 年需要重新看 MCP

2024 年底发布时，MCP 常被理解成“给模型接工具的 USB-C”。这个比喻解释了接口统一，却隐藏了最难的一半：USB 设备传输的是字节，MCP Server 暴露的能力可能读取私有仓库、运行命令、发送邮件或修改业务数据。连接一旦成功，模型生成的参数便可能跨过原来由页面、服务端校验和人工操作共同构成的边界。

生态规模已经让这个问题无法停留在试验阶段。Anthropic 在 2025 年 12 月将 MCP 捐赠给 Linux Foundation 旗下 `Agentic AI Foundation`。公告披露当时已有 10,000 多个活跃公共 MCP Server，Python 与 TypeScript SDK 每月下载量超过 9,700 万，ChatGPT、Cursor、Gemini、Microsoft Copilot 和 Visual Studio Code 均已采用。协议治理从单一厂商转向开放基金会，跨客户端复用开始形成真实收益。[2]

同一时间，安全研究开始从“提示注入可能有风险”进入具体攻击面。NSA 在 2026 年 5 月发布的 MCP 安全设计建议指出，MCP 的扩张速度已经超过安全模型的成熟速度。报告列出的风险覆盖访问控制、序列化、审批流程、Token 生命周期、工具命名冲突、下游输出投毒和审计缺失。[3] 这组事实给团队的信号很清楚：MCP 已经值得作为长期接口建设，也必须按照生产系统的标准治理。

## 核心内容一：MCP 标准化了什么

MCP 2025-11-25 规范使用 JSON-RPC 2.0 消息，定义 Host、Client、Server 三种角色。Host 是承载模型和用户交互的应用；Client 是 Host 内与某个 Server 维持连接的协议组件；Server 提供上下文和可执行能力。[1]

Server 侧能力分成三类：

- Resources 提供数据和上下文，例如文件、数据库记录或知识库条目。
- Prompts 提供可复用的消息模板和工作流入口。
- Tools 提供模型可执行的函数。

Client 也可以向 Server 提供 Sampling、Roots 和 Elicitation。Sampling 允许 Server 请求 Host 发起模型调用；Roots 告知 Server 可操作的 URI 或文件边界；Elicitation 请求用户补充信息。协议还定义能力协商、进度、取消、错误与日志等通用语义。

这套设计最大的工程价值是把“能力是什么、如何发现、如何调用”从具体智能体产品中剥离。同一个代码仓库查询 Server 可以接入桌面助手、IDE 和自动化平台；同一个 Host 也能通过一致生命周期加载本地 stdio Server 与远程 Streamable HTTP Server。客户端数量为 N、系统数量为 M 时，团队不再维护 N×M 套私有适配。

协议没有定义完整的业务事务。它不会替 Server 判断一次付款能否重试，不会替 Host 决定某个用户能否读取全部仓库，也不会保证工具描述可信。规范明确指出 Tools 可能代表任意代码执行，工具描述应作为不可信信息处理；用户同意、数据隐私与调用授权需要实现方落实。[1]

## 核心内容二：生产系统的四条边界

### 1. 能力边界：Resources 与 Tools 分开治理

读取和执行应进入不同权限面。只做调查、规划或审查的模式通常只需要 Resources。写文件、发消息、执行 SQL 和触发发布属于 Tools，需要更强授权、参数校验和审计。把所有数据访问都包装成 Tool，会失去静态只读策略；把副作用伪装成 Resource，则会让安全评审得到错误结论。

MimiAgent 当前的做法提供了可讨论样板：Plan 模式不连接 MCP Server Tools，只保留显式的 Resource 列举与读取包装；非 owner/system 的外部事件直接禁用 MCP。能力限制落在工具选择层，提示词被绕过时也无法扩大权限。这比在 system prompt 中写“不要修改数据”更接近可验证的控制。

### 2. 信任边界：连接授权不等于每次动作授权

本地 stdio Server 可以启动进程，远程 Server 可以访问组织数据。安装、连接、发现工具、调用工具是四个不同动作。生产 Host 至少要记录 Server 来源、版本或文件身份、传输方式、允许的环境变量、可见工具集和授权人。

工作区配置尤其容易混淆“仓库里的配置”与“用户做出的授权”。陌生仓库可以提交一份 `mcp.json`，其中的 stdio 命令仍然是外部输入。合理路径是将工作区真实路径和显式信任绑定，未授权时只显示诊断，不启动进程。远程凭证继续保存在环境变量或受保护的凭证系统中，不能写进可提交配置。

每次动作是否需要人工确认取决于风险，而非是否使用 MCP。读取公开文档可以自动执行；删除数据、发送外部消息、支付和生产发布需要明确审批或受控策略。NSA 报告指出，已信任 Server 的能力变化经常不会触发重新授权，这会让一次性同意悄然变成长期宽权限。[3] 团队应对工具清单或权限范围变化做 diff，并要求重新授权高风险变化。

### 3. 事务边界：超时不是失败

一次 MCP Tool 调用可能出现四种结果：明确成功、明确失败、尚未执行、结果未知。网络超时和进程崩溃通常属于“结果未知”。如果工具已经发送邮件，Client 在收到响应前断开，自动重试会产生第二封邮件。单纯增加重试次数会把可靠性问题变成重复事务。

副作用工具需要稳定幂等键、执行账本或业务侧去重。账本至少记录执行作用域、工具名、规范化参数摘要、状态和可重放结果。相同调用再次出现时，明确成功的结果可以直接返回；参数冲突应拒绝；结果未知时停止自动重放，转为查询业务状态、执行不会重复原事务的补偿动作，或向用户报告。

幂等也有边界。读取天气可以安全重试，向不支持幂等键的第三方发送消息则不能靠 Host 猜测。工具契约应该显式标注 `readOnly`、`idempotent`、`destructive`、超时策略和结果确认方式。MCP 的调用协议保持通用，事务保证由 Host、Server 和下游业务共同完成。

### 4. 数据边界：工具输出仍是不可信输入

智能体调用搜索、邮件、网页或代码托管工具后，返回内容经常再次进入模型上下文。攻击者可以把指令放在 Issue、网页、邮件正文甚至 Tool 描述中，引导模型调用其他工具或泄漏数据。MCP 让工具组合更容易，也让跨 Server 的间接提示注入更容易传播。

处理原则是保留来源并限制传播。外部内容作为 data 进入上下文，不能拼接成 system instruction；工具 A 的输出不能自动取得工具 B 的调用权限；敏感资源读取与外部写入之间设置策略门；日志记录调用链、来源 Server、参数摘要、授权决策和结果状态。多租户场景还要把 Session、用户身份和 Server 凭证绑定，避免共享上下文串线。

## MCP、内置 Tool 与 Connector 怎么分工

团队容易把所有外部能力都归入 MCP。更稳妥的选型标准是部署与凭证边界。

| 能力形态 | 推荐承载方式 | 判断依据 |
|---|---|---|
| 高频、通用、与 Host 安全模型紧密耦合 | 少量内置 Tool | 需要稳定能力元数据与统一权限策略 |
| 可跨多个智能体客户端复用的数据或工具 | MCP Server | 标准发现和调用带来明显复用收益 |
| 持有渠道 SDK/本机登录态、需要独立重启与投递确认 | 隔离 Connector | 凭证和进程生命周期不应进入 Runtime |
| 低频领域流程 | Skill 调用现有 Tool/MCP | 用说明和脚本组合能力，减少 Runtime 表面 |

以邮件为例，查询邮件可以由隔离 Connector 暴露受控 action，也可以在组织已有远程邮件服务时封装成 MCP Server。若能力直接依赖本机 Apple Mail 登录态，Connector 更容易守住凭证与应用权限边界。MCP 可以作为 Host 到企业工具平台的通用协议，不能消除隔离进程存在的理由。

## 实践建议：用两周建立 MCP 接入基线

第一周先盘点能力，不急着增加 Server。为每个现有或计划接入的 Tool 建一行清单：来源、传输、读写属性、数据级别、凭证位置、权限粒度、幂等方式、超时后处理、审计字段和负责人。没有结果确认机制的写工具标为高风险，默认不自动重试。

随后建立 Host 侧最小控制面：配置 schema 校验；工作区配置显式信任；环境变量 allowlist；只连接成功且通过策略的 Server；按运行模式和事件来源过滤 Tools；Resources 与 Tools 分离；外部内容保留 provenance；高风险动作支持批准、编辑和拒绝。

第二周做故障与攻击演练。至少覆盖五个场景：工具成功后连接中断、同一 call ID 参数变化、恶意 Tool 描述、私有资源经另一 Server 外发、Server 升级后新增写能力。验收结果应落到可观察事实：是否重复执行、是否阻断越权、日志能否还原调用链、用户能否理解授权对象。

最后再谈生态复用。优先把稳定、跨客户端需求明确、权限边界清楚的能力做成 MCP Server。只服务一个本地工作流且包含敏感凭证的能力，可以继续放在 Connector 或内置边界。协议层保持兼容，运行时保持克制，业务事务保持明确，这三项同时成立，MCP 才会从演示接口变成生产基础设施。
