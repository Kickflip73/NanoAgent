# 2025—2026 AI 前沿技术调研与组内分享选题

写作日期：2026-07-16

## 调研结论

当前最值得组内投入讨论的变化是智能体工程开始形成清晰分层：模型负责推理，运行时负责状态与可靠性，MCP 负责连接工具和数据，RAG 负责把组织知识变成可验证上下文。框架名称仍在快速变化，协议边界、执行语义和评测方法更适合作为长期技术资产。

本次从五个方向各筛选 2—3 份 2025—2026 年的一手资料或研究论文。结合 MimiAgent 当前已有的 OpenAI Agents SDK、MCP Manager、Session/Goal/Checkpoint、ExecutionLedger、RAG 与受控 Team 能力，推荐三个分享主题：

| 优先级 | 主题 | 与当前工作的关系 | 讨论价值 |
|---|---|---|---|
| P0 | MCP：智能体时代的连接层与安全边界 | 项目已支持 stdio、Streamable HTTP、Resources、Tools 与执行账本 | 决定连接器边界、权限模型和生态兼容性 |
| P1 | 可恢复的智能体运行时：框架选型之外的工程问题 | 项目已有 Session owner、Checkpoint、Goal、Plan、幂等账本与 Daemon | 可直接审视长任务、重试、副作用和多智能体的设计 |
| P1 | `Agentic RAG`：从固定检索到按需推理 | 项目已有本地混合检索，适合设计渐进式升级 | 能讨论 GraphRAG 的投入拐点及 `Agentic RAG` 的成本边界 |

推理部署和 AI 编程工具同样重要，但前者与当前仓库的产品主线距离稍远，后者更适合在团队形成实际使用数据后单独复盘。本次保留完整素材，暂不进入三篇主分享。

## 一、AI 智能体与多智能体系统

### 来源 1：OpenAI 发布 Responses API 与 Agents SDK

- 来源：OpenAI，2025-03-11
- 链接：https://openai.com/index/new-tools-for-building-agents/
- 核心内容：Responses API 把模型调用与 web search、file search、computer use 等工具统一到一个原语中；Agents SDK 提供智能体、Handoff、Guardrail 和 Tracing。它的设计倾向是保持编排层轻量，把多智能体表达为受控转交或 Agent-as-tool，而非引入完整工作流平台。
- 可引用角度：轻量 SDK 的价值在于降低模型—工具循环的样板代码，生产可靠性仍需要应用侧状态、权限、幂等和恢复机制。

### 来源 2：LangGraph 的持久化运行时

- 来源：LangChain 官方文档，2026 年访问
- 链接：https://docs.langchain.com/oss/python/langgraph/overview
- 核心内容：LangGraph 将自己定位为长时间、有状态智能体的低层编排运行时，重点能力包括 durable execution、streaming、human-in-the-loop 和 persistence。Checkpoint 可支持故障恢复、时间旅行调试和人工审批。
- 可引用角度：当业务流程有显式状态图、跨节点恢复和人工中断需求时，LangGraph 的价值来自运行时语义，不只是“画图编排”。

### 来源 3：CrewAI 的 Crews 与 Flows

- 来源：CrewAI 官方文档，2026 年访问
- 链接：https://docs.crewai.com/
- 核心内容：CrewAI 用 Crews 表达角色化协作，用 Flows 表达更确定的事件驱动控制，并提供部署、监控和 RBAC 能力。它适合快速描述“谁负责什么”，但生产系统仍要明确状态所有权、失败语义和工具权限。
- 可引用角度：多智能体的角色抽象适合原型与组织表达，流程控制与可靠执行应落在更确定的层次。

### 阶段判断

框架选型已经转向三个运行问题：谁拥有状态、失败后如何恢复、副作用如何审计。OpenAI Agents SDK、LangGraph、CrewAI 对应轻量模型循环、持久图运行时、角色化协作三种不同抽象层，功能清单无法替代运行语义比较。

## 二、RAG 架构演进

### 来源 1：Agentic RAG 综述

- 来源：Aditi Singh 等，2025-01-15
- 链接：https://arxiv.org/abs/2501.09136
- 核心内容：传统 RAG 的固定流水线难以处理多步推理与动态任务；`Agentic RAG` 引入规划、反思、工具使用和多智能体协作，由控制层动态选择检索策略并迭代修正上下文。
- 可引用角度：Agentic RAG 的新增成本来自控制循环与评测复杂度，应只用于查询分解、跨源检索或证据不足时的升级路径。

### 来源 2：Microsoft GraphRAG 的 DRIFT Search

- 来源：Microsoft GraphRAG 官方文档，2026 年访问
- 链接：https://microsoft.github.io/graphrag/query/drift_search/
- 核心内容：DRIFT 将全局社区摘要与局部搜索组合，先形成 broad answer 和后续问题，再沿局部证据迭代。目标是在全局理解、细节覆盖与查询成本之间取平衡。
- 可引用角度：GraphRAG 的优势集中在关系密集、跨文档、多跳和全局归纳问题，不应替换所有向量检索。

### 来源 3：RAG 与 GraphRAG 的系统比较

- 来源：Han 等，2025-02-17
- 链接：https://arxiv.org/abs/2502.11371
- 核心内容：论文在问答与查询式摘要等任务上系统比较 RAG 和 GraphRAG，强调图结构收益具有任务依赖性。图构建、实体关系质量和查询方式会直接改变结果。
- 可引用角度：GraphRAG 是带有额外索引成本的数据建模选择，需要用任务集验证，不能根据概念先进程度直接采用。

### 阶段判断

RAG 的演进方向是“分层路由”：基础查询继续走词法与向量混合检索；关系问题进入 GraphRAG；证据不足或查询需要拆解时才进入 Agentic loop。HyDE 仍可作为查询扩展组件，但它不是 2025 年的新架构，也不应被包装成独立终态。

## 三、LLM 推理优化与部署

### 来源 1：vLLM V1 架构升级

- 来源：vLLM 团队，2025-01-27
- 链接：https://vllm.ai/blog/2025-01-27-v1-alpha-release
- 核心内容：V1 重构核心架构，统一调度与执行路径，引入 `torch.compile` 和分段 CUDA Graph，并把多模态模型视为一等公民。优化重点从单一 kernel 转向调度、缓存、编译和模型覆盖的整体协同。
- 可引用角度：推理引擎选型应同时看 TTFT、TPOT、吞吐、并发形态、模型支持和运维复杂度。

### 来源 2：SGLang 与 ModelOpt 的量化集成

- 来源：NVIDIA ModelOpt / SGLang 团队，2025-12-02
- 链接：https://www.lmsys.org/blog/2025-12-02-modelopt-quantization/
- 核心内容：SGLang 原生接入 NVIDIA Model Optimizer，使全精度模型到量化服务的流程收敛到同一套工具链；其公开测试中，NVFP4 相对 FP8 基线可获得最高约 2 倍单卡吞吐。
- 可引用角度：量化收益必须与具体模型、硬件、精度和流量形态一起测，公开峰值不能直接等价为业务收益。

### 来源 3：DeepSeek-R1 的蒸馏模型

- 来源：DeepSeek-AI，2025-01
- 链接：https://arxiv.org/abs/2501.12948
- 核心内容：DeepSeek-R1 发布了基于 Qwen 与 Llama 的 1.5B—70B 六个蒸馏模型，展示将强推理模型能力迁移到更小模型的可行路径。
- 可引用角度：蒸馏解决能力与成本的结构性权衡，量化解决同一模型的存储和计算效率，两者目标不同，可以叠加但需要分别评测质量损失。

### 阶段判断

2025 年后的推理优化已经进入系统协同阶段：模型压缩、KV Cache、连续批处理、Prefill/Decode 调度、编译和硬件特性共同决定成本。组内若没有自托管大模型的明确体量与隐私需求，当前更适合建立评测方法，而非直接建设推理平台。

## 四、AI 编程工具

### 来源 1：Codex 的训练与沙箱

- 来源：OpenAI，2025-05-16
- 链接：https://openai.com/index/o3-o4-mini-codex-system-card-addendum/
- 核心内容：codex-1 基于真实软件工程任务进行强化学习，目标包括遵守指令、贴近 PR 偏好并迭代运行测试。云端编码智能体在临时隔离环境中工作，执行面和审查面是产品能力的一部分。
- 可引用角度：代码模型只决定能力上限，仓库上下文、沙箱、测试反馈和变更审查共同决定可交付性。

### 来源 2：GitHub Copilot Coding Agent

- 来源：GitHub，2025-05-19
- 链接：https://github.com/newsroom/press-releases/coding-agent-for-github-copilot
- 核心内容：Coding Agent 以 Issue 为入口，在后台执行任务、持续提交到 Draft PR，并通过 session log 和 PR review 接受反馈。它把智能体执行嵌入已有 SDLC 控制面。
- 可引用角度：异步 coding agent 的核心交付单位是可审查 PR，不是聊天答案。

### 来源 3：Cursor 2.0 与 Composer

- 来源：Cursor，2025-10-29
- 链接：https://cursor.com/blog/2-0
- 核心内容：Composer 在训练中使用代码库级语义搜索工具；Cursor 用 git worktree 或远程机器隔离并行任务，并把代码审查、测试和浏览器验证视为新的瓶颈。
- 可引用角度：多 Agent 编程的前提是上下文质量、路径隔离和验证吞吐，简单增加并发数会把瓶颈转移给评审者。

### 阶段判断

AI 编程工具正在从 IDE 补全演进为“任务—隔离环境—测试—PR—评审”的异步执行系统。团队落地应优先改造仓库可理解性和验证闭环，再比较模型榜单。

## 五、MCP 及其生态

### 来源 1：MCP 2025-11-25 规范

- 来源：Model Context Protocol 官方规范，2025-11-25
- 链接：https://modelcontextprotocol.io/specification/2025-11-25
- 核心内容：MCP 以 JSON-RPC 2.0 为基础，定义 Host、Client、Server 三类角色。Server 可暴露 Resources、Prompts、Tools，Client 可提供 Sampling、Roots、Elicitation；协议还覆盖能力协商、进度、取消、错误与日志。
- 可引用角度：MCP 标准化的是能力发现和调用协议，不负责业务事务、细粒度授权、幂等与审计的完整实现。

### 来源 2：MCP 捐赠给 `Agentic AI Foundation`

- 来源：Anthropic，2025-12-09
- 链接：https://www.anthropic.com/news/donating-the-model-context-protocol-and-establishing-of-the-agentic-ai-foundation
- 核心内容：MCP 被捐赠给 Linux Foundation 旗下 AAIF。公告披露当时已有 10,000+ 活跃公共 Server、Python 与 TypeScript SDK 月下载量超过 9,700 万，并被 ChatGPT、Cursor、Gemini、Microsoft Copilot、VS Code 等采用。
- 可引用角度：MCP 已从单厂商接口进入跨平台基础协议阶段，生态兼容性开始产生长期价值。

### 来源 3：NSA 的 MCP 安全设计建议

- 来源：美国国家安全局 AISC，2026-05
- 链接：https://media.defense.gov/2026/Jun/02/2003943289/-1/-1/0/CSI_MCP_SECURITY.PDF
- 核心内容：报告指出 MCP 的扩张快于安全模型建设，风险包括动态工具调用、隐式信任、上下文泄漏、授权缺失、序列化、审批流程、Token 生命周期、审计不足、工具命名冲突和下游提示注入。
- 可引用角度：生产级 MCP 需要在 Host 与业务系统侧补齐身份、最小权限、输入验证、沙箱、审批、幂等和全链路审计。

### 阶段判断

MCP 值得重点投入，因为它正在形成智能体连接层的事实标准，同时仍把安全与事务责任留给实现方。分享需要回答：哪些能力做 MCP Server，哪些留在 Connector；信任如何授予；未知结果如何处理；工具输出怎样作为不可信数据继续流转。
