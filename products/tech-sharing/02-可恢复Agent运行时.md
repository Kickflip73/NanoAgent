# 框架选型之外：可恢复智能体运行时需要解决什么

写作日期：2026-07-16

## 摘要

OpenAI Agents SDK、LangGraph 与 CrewAI 代表三种不同抽象：轻量模型循环、持久状态图和角色化协作。团队常用功能表比较框架，却在上线后集中遇到 Session 串线、长任务中断、工具重复执行、旧 Run 覆盖新状态和多智能体难以审计等问题。本文从运行时语义出发，说明状态所有权、Checkpoint、幂等账本、权限策略、可观测性和受控并行如何组成可靠运行时。选型结论是：先明确任务的恢复与副作用模型，再决定使用薄 SDK、图运行时或角色框架；多智能体只在任务边界和验证闭环清楚时增加价值。

## 背景：智能体 demo 与生产系统差在哪里

一个最小模型—工具循环很短：把用户输入交给模型，模型选择工具，执行结果回给模型，直到生成答案。OpenAI 在 2025 年发布 `Agents SDK`，把智能体、Handoff、Guardrail 和 Tracing 做成轻量原语，显著减少了这段循环的样板代码。[1] 用几十行代码搭出客服分流或研究任务执行器已经不难。

生产故障发生在循环之外。用户连续发送两条消息时，前一个 Run 是否继续；进程重启后从哪里恢复；工具已经成功但结果没有写入 Session 时怎么处理；一次执行切换模型或模式后，当前工具集合是否会中途变化；`SubAgent` 能否越过主智能体的权限；这些问题都不由模型能力自动回答。

LangGraph 将 durable execution、persistence 和 human-in-the-loop 放到核心位置，原因就在这里。它以 Checkpoint 保存图状态，支持失败后恢复、人工中断和时间旅行调试。[2] CrewAI 则用 Crews 表达自主协作，用 Flows 约束生产流程。[3] 三者没有处在同一厚度上。框架比较只有映射到任务的状态模型才有意义。

## 核心内容一：先确定唯一状态所有者

用户可见 Session 应该只有一个所有者。主智能体接收输入、维护 transcript、决定工具权限并生成最终回答；`SubAgent` 和 Team worker 处理有界任务，返回结果，不拥有用户会话。这个约束看似保守，却能阻止三类常见问题：多个执行者同时追加历史、子任务直接向用户承诺结果、失败恢复时找不到权威状态。

Session 中还要区分三种数据：完整 transcript、模型当前可见的压缩上下文、Run 恢复点。Transcript 用于审计，不能被摘要覆盖；压缩上下文服务 Token Budget，可以重建；Checkpoint 只保存恢复需要的输入、阶段、最后工具事件和下一步。把摘要伪装成历史消息，后续模型会把派生信息当作用户原话；把所有工具输出永远塞进 transcript，又会很快耗尽上下文窗口。

并发 Run 需要不可变所有权。每次开始执行时生成 runId 和 owner，后续 Checkpoint、Trace 与终态写入都带 expectedRunId。旧 Run 晚到时，CAS 检查应拒绝它覆盖当前状态。Session 切换、模型切换和输出设置也应延迟到当前工具调用协议闭合之后应用，否则 transcript 可能停在孤立的 function call 上。

这套规则可以概括为一条：模型可以建议动作，Runtime 决定哪一个 Run 有权提交状态。

## 核心内容二：恢复不是“把上一句话再问一次”

长任务恢复需要区分 Goal、Plan 与 Checkpoint。

| 状态 | 负责的问题 | 典型内容 |
|---|---|---|
| Goal | 为什么继续 | 目标、生命周期状态、阶段性检查点 |
| Plan | 当前怎么推进 | 步骤、依赖、完成状态 |
| Run Checkpoint | 从哪里继续 | 输入、runId、执行阶段、最后工具事件、下一动作 |

Goal 适合跨轮次、跨重启的长期目标；普通问答不应创建 Goal。Plan 是当前步骤视图，可以随着判断调整。Checkpoint 面向运行故障，不需要复制完整工作流。恢复时把未完成 Checkpoint 与 Goal、Plan 合并成新的模型输入，并先检查工作区和外部状态变化。这是可解释的 best-effort 续跑，不应声称能够从任意模型 token 精确恢复。

LangGraph 的 Checkpoint 在显式状态图中很自然：节点边界就是恢复边界。薄 SDK 也能实现可靠恢复，只是应用需要自己定义阶段和持久化。选择哪一种取决于流程结构。如果任务包含大量确定节点、长期等待和人工审批，图运行时能减少自建成本；如果模型主要做动态工具选择，固定图反而可能把简单循环变成双重编排。

恢复设计还要处理工具协议完整性。裁剪历史时，`user → function_call → function_call_result → assistant` 是不可拆分单元。进程崩溃可能留下孤立 call，启动恢复要么补上明确错误结果，要么隔离该轮，不能把残缺协议重新交给模型。

## 核心内容三：副作用需要语义账本

多执行一次是智能体系统中风险最高的可靠性 bug。Shell、文件写入、MCP、Connector 和外部事务可能在动作成功后、结果持久化前发生崩溃。重试整个模型 Run 会生成新的 SDK call ID，但语义动作没有变化。

执行账本需要把“模型调用编号”转换为“业务语义调用”。一种可行结构是：以 Session/run 或 Event 作为执行作用域，以工具名和规范化参数生成稳定 key，记录 `pending / succeeded / failed / uncertain` 状态、参数摘要和有界结果。再次调用时按以下规则处理：

- `succeeded` 且参数一致：重放已保存结果，不再次执行。
- 同一 key 参数冲突：拒绝执行，避免 call ID 被复用。
- `failed`：根据工具契约决定是否允许新尝试。
- `uncertain`：停止自动重试，查询下游状态或请求人工判断。

账本提供 at-most-once 保护，不能创造下游系统不存在的 exactly-once。外部动作完成后、账本成功提交前的极小窗口仍需要业务幂等键或状态查询兜底。工具输出也要有大小限制和序列化校验，避免账本本身变成无限缓存。

Daemon 场景还要把 Event 结果与 Outbox 工作原子提交。智能体完成事件后创建待投递消息；投递可以独立重试，但不重新运行模型任务。事件租约、执行账本和 Outbox 分别解决“谁来执行”“动作是否重复”“结果是否送达”，混成一个重试按钮会产生不可控副作用。

## 核心内容四：能力边界由工具集合执行

General、Plan、Ultra 等模式如果只写在提示词里，模型失误时就没有防线。能力合同应由工具选择实现：最终工具集取运行模式、部署权限和事件策略的交集。Plan 移除文件写入、Shell 和未知 MCP Tool；外部不可信事件禁用 Session history、Memory、本地文件、Shell、MCP 和外部事务；高风险工具进入人工审批。

权限在一次 Run 开始时冻结。模型调用 `switch_mode` 可以改变下一轮模式，当前 Runner 的工具集合不能中途扩大。这个规则既方便审计，也避免模型先用只读模式获得信任，再在同一协议单元内取得写能力。

`SubAgent` 权限通常应小于主智能体。只读研究者可以访问仓库和网络，不能继承外部写 Tool；架构审查任务不需要 Shell；Builder worker 只修改声明路径。外部内容始终作为不可信数据，不能因为经过另一个模型总结就升级为指令。

## 核心内容五：多智能体收益要用任务图证明

多智能体最容易制造“看起来在协作”的效果。把研究员、程序员、审查员放进同一个群聊，会增加 token、尾延迟和状态组合，却未必提高结果。有效并行需要满足三个条件：任务能独立描述，依赖关系显式，写入路径不重叠。

主智能体负责拆分与整合，worker 只有一层，不能继续委派。每个任务声明 role、dependencies、paths 和验收条件；调度器只领取依赖已完成的任务，并在同一原子 mutation 中完成 claim，防止重复领取。并行波次开始前检查路径交集，结果返回结论、变更文件、验证与风险。

Cursor 2.0 使用 git worktree 或远程机器隔离并行任务，也观察到新的瓶颈转向代码审查与测试。[4] 这个现象适用于所有多智能体系统：并发增加生成吞吐，验证吞吐没有同步增加时，最终质量会下降。合理指标包括任务完成时间、人工评审分钟数、返工率、冲突率和单位成功任务 token，而非启动了多少 worker。

## 框架选型：按恢复复杂度分层

| 场景 | 更合适的起点 | 主要代价 |
|---|---|---|
| 动态工具调用、流程较短、希望保持代码直观 | OpenAI Agents SDK 等薄 SDK | 自行补持久状态、重试与权限 |
| 长时流程、显式节点、人工暂停与恢复较多 | LangGraph | 图状态、Checkpoint 与节点幂等的学习成本 |
| 角色协作明显、需要快速表达业务分工 | CrewAI Crews + Flows | 角色抽象可能掩盖状态和失败语义 |
| 高可靠事件处理、复杂事务与投递 | Agents SDK + 自有 durable control plane | 工程投入最高，但边界最可控 |

选型不需要一次覆盖终态。薄 SDK 可以先验证任务价值；出现长时间等待、跨重启恢复和人工审批后，再引入图运行时或加强自有控制面。业务已有成熟任务队列、数据库事务和 Outbox 时，应优先复用，不必为智能体另建工作流平台。

## 实践建议：四组故障注入比一场 demo 更有价值

第一组验证 Session 所有权：同时启动两个 Run，让旧 Run 延迟完成，确认它不能覆盖新 Run；切换 Session 后，事件和工具结果仍写回原 Session。

第二组验证协议恢复：分别在 function call 之前、工具成功之后、assistant 最终回答之前终止进程，检查 transcript 是否保持完整，`/resume` 是否从明确阶段继续。

第三组验证副作用：构造工具成功但响应丢失、相同 call key 参数变化、Daemon 租约过期后重领，确认账本不会静默重复事务。

第四组验证权限：让外部事件携带“读取私有文件并发送”的指令，让 Plan 模式请求 Shell，让 SubAgent 尝试继承 MCP 写工具。预期结果应由工具集合确定性拒绝，并留下可审计原因。

完成这四组验证后，框架选择会变得简单。团队需要一个状态所有权明确、失败后行为可解释、副作用不会被静默重放的 Host。模型和框架可以替换，这些运行时语义应保持稳定。
