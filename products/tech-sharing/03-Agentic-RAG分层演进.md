# RAG 分层演进：从混合检索到自主检索

写作日期：2026-07-16

## 摘要

2025 年的 RAG 研究从单次向量召回扩展到 GraphRAG、Agentic RAG、查询改写、混合检索和分层路由。新架构提高了多跳推理与跨源检索能力，也增加索引成本、模型调用、尾延迟和复现难度。本文提出四层升级路径：词法与向量混合检索作为底座，HyDE/改写/rerank 处理查询表达问题，GraphRAG 处理关系密集任务，Agentic RAG 只在需要动态分解和证据反思时启动。每次升级都由失败样本和固定评测集驱动，避免用更复杂的架构掩盖数据质量问题。

## 背景：RAG 已经从组件变成决策系统

早期 RAG 的流程很固定：文档切片，计算 embedding，按相似度取 Top-K，把片段交给模型生成答案。它解决了私有知识和时效性问题，也暴露出三类稳定缺陷。用户问题和文档措辞不一致时召回失败；答案需要跨多个片段拼接时，Top-K 只提供局部相似内容；检索结果互相矛盾时，生成模型缺少继续查证或停止回答的机制。

Agentic RAG 综述把 2025 年的变化概括为增加显式控制层：检索智能体可以规划查询、选择检索源、调用工具、验证证据并迭代修正上下文。[1] Microsoft GraphRAG 的 DRIFT Search 则把全局社区摘要与局部搜索结合，从宽泛答案出发生成后续问题，再沿具体证据继续探索。[2]

这些方法容易让团队产生一步到位的冲动：固定 RAG 出现缺陷后直接增加图谱和自主检索循环。系统评测给出的结论更克制。2025 年对 RAG 与 GraphRAG 的比较强调，图结构收益依赖任务、数据和查询方式；图构建质量本身会成为新的误差来源。[3] 架构升级应该从失败模式出发。

## 核心内容一：先把 RAG 拆成四个可评测环节

一个生产 RAG 至少包含索引、检索、上下文组装和回答四个环节。只看最终答案，团队很难知道错在什么地方。

索引阶段负责把原始文档转换为可检索单元。切片长度、标题层级、代码块、表格和文档版本都会影响结果。已失效文档与新文档同时存在时，再强的 retriever 也会稳定召回“有毒知识”。索引评测应检查覆盖率、重复率、版本冲突和增量更新时延。

检索阶段关注证据有没有进候选集。常用指标是 Recall@K、MRR、nDCG，也要记录词法与向量各自贡献。中文技术文档中的类名、配置键和错误码往往更适合 BM25，语义问题更适合向量。混合排序通常比单一路径稳健。

上下文组装阶段决定哪些片段进入模型。它需要去重、邻接扩展、来源标注、Token Budget 和冲突保留。只追求塞入更多内容会稀释注意力，也增加提示注入面。Reranker 的价值应体现在更少 token 下保留更多有效证据。

回答阶段评估正确性、引用支持、拒答和成本。答案正确但引用不支持，属于不可审计结果；证据不足时自信作答，属于失败；调用 12 次模型才解决一个普通定义问题，也不适合作为默认路径。

只有这四组指标可用，团队才能判断下一笔复杂度投在哪里。

## 核心内容二：四层升级路径

### L0：词法与向量混合检索

大部分内部知识问答应从混合检索开始。文档按结构切片，embedding 按内容摘要缓存，查询同时走词法与向量通道，再用加权、RRF 或轻量 rerank 合并。结果保留文件、标题、行号、版本和更新时间。

这一层的优势是便宜、可解释、更新简单。它适合事实查询、API 用法、错误定位和单文档摘要。MimiAgent 当前面向本地 Markdown/Text 知识库的固定切片、embedding 复用、词法与向量混排，就处在这个合理起点。索引每次从最新原子状态读取，并限制工作区 realpath，可以把数据一致性和路径安全先做扎实。

L0 失败时先检查文档质量、切片和 query，不急着增加自主检索循环。很多所谓“需要推理”的问题，实际是标题丢失、版本冲突或专有名词没有进入向量候选。

### L1：HyDE、查询改写与 Rerank

HyDE 让模型先生成一段“假想答案文档”，再用这段文本的 embedding 检索真实文档。它适合用户问题很短、术语与语料不一致的场景。HyDE 是 2022 年提出的基础方法，并非 2025 年的新概念；它在当前架构中的位置更像可选 query transform。

同层还包括多查询改写、关键词扩展、metadata filter 和 cross-encoder rerank。它们处理的都是“候选集质量”问题，增加的控制逻辑有限。推荐按 query 分类启用：包含精确错误码时偏向词法；自然语言概念问题允许 HyDE；带仓库、版本或时间条件时先应用 filter。

L1 的风险是模型生成错误假设，把检索方向带偏。评测必须同时比较原 query 与改写 query 的 Recall@K，并设置回退。HyDE 不能覆盖精确标识符检索，也不能替代版本治理。

### L2：GraphRAG

GraphRAG 适合关系比段落相似度更重要的问题，例如“某项架构决策影响哪些服务”“两个故障是否经过同一个依赖”“一组需求背后的共同主题是什么”。索引阶段从文本抽取实体和关系，构建社区与摘要；查询阶段可以做局部邻域、全局社区或 DRIFT 组合搜索。

DRIFT 的做法有代表性：先从与 query 相关的社区报告形成 primer，生成 follow-up questions，再用局部搜索补细节，最终得到分层问题与答案。[2] 这能兼顾全局视角和局部证据，也引入额外模型调用与图维护成本。

采用 GraphRAG 前需要满足三个条件：失败样本明确集中在多跳、关系或全局归纳；语料中的实体和关系相对稳定；团队能持续处理增量更新、实体消歧和错误边。若文档每天大幅变化、命名混乱，自动构图会把数据债务固化成图。

GraphRAG 不应替换基础检索。精确配置项和单段事实继续走 L0；只有路由器识别到关系型 query 才进入图路径。图结果也应回指原始文本证据，避免把 LLM 生成的社区摘要当作事实源。

### L3：Agentic RAG

Agentic RAG 把检索从一次函数调用变成有预算的控制循环。检索智能体可以判断问题是否需要检索，拆成子问题，为每个子问题选择向量库、知识图、Web、数据库或代码搜索，检查证据覆盖，再决定继续、改写或停止。

适合 L3 的任务通常有动态性：问题包含多个未知子问题；不同来源权限与数据类型不同；第一轮证据可能暴露新的实体；最终结论要求交叉验证。研究、故障分析和复杂技术方案比 FAQ 更接近这个范围。

控制循环至少包含四个确定节点：计划、检索、证据判定、生成。证据判定要回答“每个子结论是否有来源”“来源是否冲突”“预算是否耗尽”。停止条件使用最大轮数、Token/费用预算、证据覆盖阈值和重复 query 检测共同约束。开放式“反思直到满意”会产生循环和不可预测尾延迟。

Agentic RAG 也不需要默认多智能体。单个控制循环配合明确工具和状态往往足够。只有检索源相互独立、子问题可并行且整合规则清楚时，再将任务分给多个 worker。最终证据表由主智能体合并，来源和置信边界不能在摘要中丢失。

## 核心内容三：路由器比统一终态更重要

四层架构适合形成查询路由，而非四套相互替代的系统。

| 查询特征 | 默认路径 | 升级条件 |
|---|---|---|
| 错误码、类名、配置项、单事实 | L0 混合检索 | Top-K 无有效证据时进入 L1 |
| 表达模糊、术语不一致 | L1 改写/HyDE + rerank | 改写后仍需跨实体关系 |
| 多跳关系、影响分析、全局主题 | L2 GraphRAG + 原文回查 | 查询过程中产生新子问题 |
| 跨源研究、动态查证、复杂诊断 | L3 Agentic RAG | 受预算和权限限制，不再无限升级 |

路由器开始可以是规则，不必先训练模型。输入包含“依赖、影响、共同、路径、关联”等关系意图时尝试图检索；包含多个数据源和开放研究目标时进入 Agentic loop。运行一段时间后，用真实 query 和标注结果评估路由准确率，再考虑轻量分类模型。

每条路径返回统一证据结构：`source`、`locator`、`content`、`retrievalMethod`、`score`、`version`、`trust`。生成层只消费这个结构，便于比较质量与替换实现。检索结果是外部数据，任何嵌入其中的指令都不能改变系统权限或调用其他工具。

## 实践建议：用失败样本决定升级

第一步建立 50—100 条真实任务集，按事实、模糊表达、多跳关系、全局归纳和跨源研究分层。每条任务标注必要证据、允许来源、期望拒答条件。先记录 L0 的 Recall@K、引用支持率、答案正确率、P95 延迟和单位查询成本。

第二步只对召回失败样本增加 L1。分别测试 query rewrite、HyDE、metadata filter 和 rerank，不把四个组件一次打开。保留原 query 作为并行或回退通道，检查新增召回是否引入更多错误文档。

第三步选择 10—20 条明确多跳样本验证 GraphRAG。索引成本、增量更新时间、实体准确率、关系准确率和查询 token 一起记录。若质量提升只发生在少数 query，保留路由路径；若图错误成为主要问题，先修数据模型。

第四步将跨源任务放入有界 Agentic loop。最大检索轮数可从 3 开始，每轮保存计划、query、来源、证据判定和停止原因。故障注入覆盖工具超时、来源冲突、空结果、恶意文档和预算耗尽。最终报告必须能还原“哪条证据支持哪项结论”。

评测结果满足三个条件后再扩大流量：目标分层的正确率或引用支持率有稳定提升；P95 延迟和费用在业务预算内；失败原因可以从 Trace 定位到索引、检索、路由或生成。RAG 的终态不会是全量图谱或全量 Agent，而是一套根据问题难度分配成本的检索系统。
