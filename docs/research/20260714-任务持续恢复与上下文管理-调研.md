# 任务持续恢复与上下文管理调研报告

日期：2026-07-14  
状态：已审核（用户授权设计后直接实施）

## 调研范围

- 目标：增强普通任务与 Ultra 任务跨轮、跨会话切换和进程重启后的持续性与可恢复性，并补齐分层上下文压缩。
- 涉及文件：
  - `src/index.ts`
  - `src/runtime/mimi-agent.ts`
  - `src/core/session.ts`
  - `src/core/context.ts`
  - `src/core/plan.ts`
  - `src/core/team.ts`
  - `src/commands.ts`
  - `src/terminal.ts`

## 核心发现

### 现状分析

1. CLI 队列负责多轮调度，`MimiAgent.stream()` 负责单轮 Agent/Tool Runner，已经具有两层雏形，但成功、失败和中断的生命周期收尾分散在 CLI 与 Runtime。
2. `FileSession` 仅持久化 SDK transcript。进程在工具执行中退出时，下一次启动只能删除孤立 Tool Call，无法说明上次任务、最后进展和恢复入口。
3. Goal、Plan、Team 均按 Session 持久化，但普通任务不一定创建 Goal；即使创建，checkpoint 也依赖模型主动调用 `update_goal`。
4. `/resume` 只支持 Goal。切换到一个被中断但没有 Goal 的会话时，只能回放历史，不能恢复到最新进展点。
5. `ContextManager` 会按完整用户轮次、条目数和 Token Budget 裁剪，并将旧历史临时拼成摘要；原始 Session 不受污染，这是正确的不变量。
6. 旧历史摘要每轮重新计算且不持久化；旧工具结果没有单独瘦身；没有 `/compact`；上下文状态只报告一个估算总数。

### 关键流程

```text
CLI queue（多轮）
  → MimiAgent.stream（单轮准备）
  → SDK Runner（模型/工具循环）
  → CLI 消费流
  → MimiAgent.completeRun / failRun
```

### 现有约束

- 保持单进程、本地 JSON/JSONL，不引入数据库、队列或工作流 DSL。
- 保持 `runtime / core / extensions / tools` 边界。
- 完整 transcript 必须可审计；压缩不能覆盖或伪造原始历史。
- Tool Call 与 Tool Result 必须成对保留，裁剪必须以完整用户轮次为边界。
- OpenAI 与 DeepSeek 共享同一恢复和上下文能力。

### 风险与问题

- Session 文件格式扩展必须兼容旧文件。
- 运行中崩溃留下的 `running` 状态必须在下次打开时确定性转换成 `interrupted`。
- 高频持久化每个模型 delta 会造成写放大；检查点只记录阶段和工具事件，不记录流式文本 delta。
- 自动压缩若直接删除 transcript 会破坏审计与 SDK Session，因此只改变“送给模型的有效视图”。
- 规则摘要不等同于额外模型生成的语义摘要，但无额外调用、无额外成本且 Provider 行为一致；本阶段采用结构化归档，后续可插入可选语义总结器。

## 与任务相关的关键结论

- 不复制 Claude Code 的完整隐式状态机；用一个小型 `RunCheckpoint` 表达 MimiAgent 真正需要的状态：`running / completed / interrupted / failed`。
- 检查点和 `ContextArchive` 与 transcript 同属 Session 生命周期，直接作为 Session 元数据持久化最清晰。
- `/resume` 应优先恢复未完成运行，并合并 Goal、Plan、Team，而不是强依赖 Goal。
- 上下文采用四层：旧 Tool Result microcompact、持久 context collapse、手动 full compact、最终按完整轮次 PTL truncation。
- 原始 transcript 始终完整存档；压缩档案只记录覆盖位置、结构化摘要和 Token 统计。
