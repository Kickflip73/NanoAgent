# Mimi 跨渠道人物上下文调研报告

日期：2026-07-15
状态：已审核（用户已明确要求直接开发）

## 调研范围

- 目标：让 Mimi 在邮件、IM、群聊等不同来源中识别 owner 配置的同一个人，并连续使用关系上下文与长期记忆。
- 涉及文件：
  - `src/daemon/attention.ts`
  - `src/daemon/policy.ts`
  - `src/runtime/mimi-agent.ts`
  - `src/core/memory.ts`
  - `tests/daemon-attention.test.ts`
  - `tests/daemon-policy.test.ts`
  - `tests/core.test.ts`

## 核心发现

### 现状分析

Daemon 默认用 `profile + source + actor + conversation` 生成 Session。相同人物从 Mail、Messages、大象或群聊触发时会落入不同 Session。Standing Orders 可以按单个 source/actor 匹配，但没有跨来源的 canonical person。

Memory 在 MimiAgent 注入 RunCause 说明前就用原始 input 做词法检索。检索 query 不含 actor、conversation 或人物身份；Memory 自身也只记录 Event/source/trust，因此“这个人喜欢什么、上次答应了什么”无法稳定跨渠道召回。

### 关键流程

`AttentionEngine.decide` 是 owner 配置与 Event 相遇的唯一确定性位置；`decideEvent` 负责 Session 和 RunCause；MimiAgent 随后检索 Memory 并构建 Prompt。因此 identity resolution 应留在 Attention Host，不应进入 Connector 或再调用一个模型。

### 现有约束

- 外部显示名和消息正文都不是可信身份；只有 owner 本机配置可以声明 alias。
- `trust` 继续只作 provenance，不转化为权限层。
- 显式 Event `sessionKey` 必须保持最高优先级。
- 配置热重载、字符上限和 status 隐私规则需要保持。

### 风险与问题

- 自动按显示名合并会把同名人物错误关联。
- 通配 alias 可能重叠，需要确定性的配置顺序和精确重复校验。
- 把私人姓名/context 全量暴露在 status 会泄露 owner 关系数据。

## 与任务相关的关键结论

在 `assistant.json` 增加最多 100 个 owner-managed people，每人包含稳定 ASCII ID、displayName、最多 20 个 `source + actor` alias 和最多 10 条可信 context。Attention 按配置顺序解析首个匹配项，使用稳定 `mimi-person-<id>` Session，把可信人物 context 与 Standing Orders 分区注入；RunCause 和 Memory 记录 person/actor/conversation provenance，Memory 检索同时匹配这些字段。无需联系人镜像、身份图、数据库迁移或额外模型调用。
