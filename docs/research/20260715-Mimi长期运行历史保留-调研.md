# Mimi 长期运行历史保留调研报告

日期：2026-07-15
状态：已审核（owner 已明确授权持续直接实施）

## 调研范围

- 目标：确认 Mimi 常驻数月或数年时 SQLite 数据是否有界，并找出不会破坏可靠性、去重和失败恢复的清理边界。
- 涉及文件：
  - `src/daemon/store.ts`
  - `src/daemon/dispatcher.ts`
  - `src/daemon/attention.ts`
  - `tests/daemon-store.test.ts`
  - `tests/daemon-attention.test.ts`

## 核心发现

### 现状分析

Mimi 的 Event、Host Run、Outbox、Digest、Audit、Schedule 和 Attention checkpoint 都持久化在 SQLite WAL。除显式删除 Schedule 外，当前没有历史清理路径。RSS、系统状态、文件活动、邮件和 IM 长期运行后会持续增加终态 Event；每次模型尝试、投递重试和注意力简报还会增加关联行。

SQLite 会复用已释放页面，但只有实际删除旧行后才能形成稳定容量。仅执行 WAL checkpoint 或 `VACUUM` 不能替代数据保留策略；频繁 `VACUUM` 还会长时间独占数据库。

### 关键引用关系

- `runs.event_id` 引用 `events.id`。
- `outbox.event_id` 引用 `events.id`。
- `digest_items.event_id` 与 `briefing_event_id` 都引用 Event。
- `audit_events.entity_id` 没有外键，但承担失败和重试诊断。
- `attention_state` 保存 briefing 日期/时点 checkpoint；`schedules` 保存一次性和周期唤醒。
- `source + external_id` 的唯一约束依赖 Event 行本身；清理后，超过保留窗口的同一外部事件理论上可以再次进入。

### 现有约束

- queued/running Event、pending/sending Outbox、未归档 Digest 和启用 Schedule 都是活状态，不可自动删除。
- dead letter 是最终失败的 owner 修复入口，Event/Outbox 及其 audit 不应被普通历史保留任务删除。
- 只有已 sent Outbox 和已 digested Digest 可以先删除，随后才能删除不再被引用的 completed/ignored/digested Event 和其 Run。
- 单 Dispatcher 已有固定轮询，不需要增加维护服务或后台线程。

### 风险与问题

- 先删 Event 会触发外键失败，或在关闭外键时留下悬挂引用。
- 清理 `attention_state` 太新记录会导致同一日简报重复；清理未归档 Digest 会丢消息。
- 永久保留所有 dedupe identity 会继续造成行数无界；有限保留意味着非常旧的来源 ID 若被 Connector 重新回放，可能再次处理。
- 自动 `VACUUM` 会阻塞常驻事务，不适合作为周期默认动作。

## 与任务相关的关键结论

在 `assistant.json maintenance` 中提供默认启用的 90 天保留期和 24 小时检查周期。Store 在一个 transaction 内按引用顺序删除：旧 sent Outbox、旧已归档 Digest、可安全删除 Event 的 Run、无引用的 completed/ignored/digested Event、旧 disabled Schedule、旧 Attention checkpoint 和不属于活状态/dead letter 的 Audit。最后执行轻量 `PRAGMA optimize` 和 passive WAL checkpoint，不自动 `VACUUM`。Dispatcher 只在 `processOnce()` 开头低频调用，不增加循环、表或权限层。
