# Mimi 文件活动雷达调研报告

日期：2026-07-15  
状态：已审核（用户已授权直接实施）

## 调研范围

- 目标：让 Mimi 持续感知 Downloads、Desktop、共享落盘目录和自动化输出中的新建/修改文件，并主动处理。
- 涉及文件：
  - `examples/connectors/file-radar-connector.mjs`
  - `mimi.files.example.json`
  - `mimi.connectors.example.json`
  - `tests/file-radar-connector.test.ts`

## 核心发现

### 现状分析

MimiAgent Runtime 已有完整文件读取、搜索、写入、移动和 Shell 工具；Daemon 也已有可靠 Event、Attention、Session 与跨重启去重。缺口不是再次实现文件操作，而是没有常驻入口能在文件落盘时主动唤醒 Mimi。

现有 Radar Connector 已证明“无回调来源使用独立、有界轮询 Connector”可以保持内核轻量。文件活动可以沿用同一模式：Connector 只扫描配置目录并输出元数据，Agent 需要时再使用现有文件工具读取或整理。

### 关键流程

```text
Downloads / Desktop / shared inbox
  -> bounded metadata scan
  -> stable file_activity Event
  -> Attention run / digest / ignore
  -> existing Runtime file tools
  -> read, summarize, move, rename or archive
```

### 现有约束

- 不读取或复制文件正文，Event 只包含路径、大小、扩展名和时间。
- 不跟随文件或目录符号链接，避免递归环和意外越过 watch root。
- 不保存第二份 cursor；`watchId + path + mtime + size` 生成稳定 externalId，由中心 Store 跨重启去重。
- 扫描深度、访问条目数、单轮事件数、回看窗口和 action 结果数都必须有界。
- 使用 Node.js 内置文件系统 API，不增加 chokidar 或工作流框架。

### 风险与问题

- macOS 的 Downloads、Desktop 或网络盘可能受系统隐私权限保护；Connector 应清晰报告单目录错误，不让其他 watch 停止。
- 轮询不能精确区分 create 与 modify，因此事件语义应诚实标为 `created_or_modified`。
- 初次启动会看到回看窗口内已有文件；稳定 ID 使重启不会重复执行已入库事件。
- 大目录必须在扫描条目上限处停止，并在 action 结果中报告 truncated，而不是无限遍历。

## 与任务相关的关键结论

1. 新增一个跨平台、零依赖的 `file-radar` Connector，不修改 Runtime/Daemon 核心。
2. 配置采用明确 watch 列表、扩展名过滤和有限递归，不实现通用 glob/规则 DSL。
3. 提供 `scan_now`、`recent_files`、`watches` 三个 action；文件实际处理继续复用现有工具。
4. Event 默认是 `ambient`，每个 watch 可显式选择 `alert` 或 `command` 及优先级，继续由 Attention 决定处理时机。
