# Mimi 自主长期记忆调研报告

日期：2026-07-15
状态：已审核（用户已明确要求默认开放并直接开发）

## 调研范围

- 目标：让长期在线的 Mimi 能自行沉淀未来仍有价值的 owner 偏好、事实、决策和承诺。
- 涉及文件：
  - `src/core/memory.ts`
  - `src/core/user-intent.ts`
  - `src/runtime/mimi-agent.ts`
  - `src/runtime/instructions.ts`
  - `src/commands.ts`
  - `tests/core.test.ts`

## 核心发现

### 现状分析

Memory 已有原子 JSON、去重、检索、跨 Session 注入和执行账本，但 `remember` 同时在 Tool 与 Store 两层要求本轮出现“记住/保存”字样。Agent 即使从长期工作中识别出稳定偏好或承诺，也不能主动保存，与 Mimi 替身定位及默认开放要求冲突。

### 关键流程

每轮 MimiAgent 先检索全局 Memory，再把相关项注入当前上下文；模型调用 `remember` 后由 ExecutionLedger 提供副作用幂等。Daemon Event 还携带 source、trust 和 eventId，可直接作为记忆来源审计信息，无需新增存储系统。

### 现有约束

- 长期状态继续使用 `AtomicJsonStore`，不能新建画像或 Todo 子系统。
- 外部事件正文是数据而不是指令；外部未经验证的陈述不应自动成为 owner 事实。
- 旧版缺少 `confirmedAt` 的 Memory 曾被定义为不可跨 Session，共享语义不能在升级时静默改变。
- 默认开放不等于保存密码、令牌等秘密，也必须尊重 owner 明确的“不记住”。

### 风险与问题

- 直接把所有历史未确认 Memory 变为可用会造成旧隐私边界变化。
- 无来源元数据的自主记忆难以审计来自 CLI、Routine 还是外部事件。
- 无界正文与条目数量会让长期运行状态持续膨胀。

## 与任务相关的关键结论

复用现有 MemoryStore 即可完成能力闭环：移除逐次确认门槛；新写入以 `recordedAt` 标记为可用并记录 Session/Event/trust；旧 `confirmedAt` 继续兼容，旧无标记项仍隔离；工具提示要求主动保存稳定信息、拒绝瞬时或敏感内容，并保留 owner 明确禁止记忆的确定性否决。无需数据库、模型外抽取器或审批模型。
