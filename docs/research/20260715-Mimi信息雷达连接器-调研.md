# Mimi 信息雷达连接器调研报告

日期：2026-07-15  
状态：已审核（用户已授权直接实施）

## 调研范围

- 目标：让 Mimi 能持续感知 RSS/Atom 新闻与关注地点天气风险，并复用已有 Attention/Briefing 和 Connector Action Bridge。
- 涉及文件：
  - `examples/connectors/radar-connector.mjs`
  - `mimi.radar.example.json`
  - `mimi.connectors.example.json`
  - `docs/CONNECTORS.md`
  - `tests/radar-connector.test.ts`

## 核心发现

### 现状分析

Daemon 已能从独立 NDJSON 子进程接收 Event，按 `source + externalId` 跨重启去重，且 Attention 已支持将 `news:*` 类信号汇总到定时简报。现有示例配置留有 weather 占位，但没有可运行实现；新闻也只有协议说明。

### 关键流程

```text
RSS / Atom / Open-Meteo HTTP
  -> radar connector bounded fetch
  -> normalized news/weather Event
  -> Mimi durable dedupe
  -> Attention run/digest/notify
  -> Agent decision and optional action
```

### 现有约束

- 保持单进程、零额外 npm 依赖，不为信息源引入独立数据库或 MQ。
- stdout 只输出 NDJSON 协议，诊断输出 stderr。
- 每次 HTTP 获取必须有超时、大小上限和有界条目数。
- RSS/Atom 条目使用 guid/id/link 作稳定身份；缺失时才对标题和时间做 hash。
- Open-Meteo 官方 Forecast API 提供小时级 `temperature_2m`、`precipitation_probability`、`weather_code` 和 `wind_gusts_10m`，可用 `forecast_hours` 限定时窗。

### 风险与问题

- 不同 RSS 实现存在 namespace、CDATA 和 HTML 差异；解析器只做订阅所需的有界子集，不尝试成为通用 XML DOM。
- 天气预报不是政府预警；Event payload 必须保留数值和触发原因，不把模型阈值命名为官方预警。
- 连接器重启会重新拉取最新条目，但 Store 的稳定 externalId 会阻止重复 Agent Run。

## 与任务相关的关键结论

1. 实现单个 `radar-connector.mjs`，仅内建 RSS/Atom 和 Open-Meteo 两类明确 driver，不设计通用爬虫 DSL。
2. 用独立 `mimi.radar.json` 定义多个 source，Connector Host 仍只需一个子进程。
3. 新闻默认是低优先级 `ambient`，交给 Attention 汇总；超阈值天气是高优先级 `alert`。
4. 提供 `refresh`、`weather_snapshot`、`sources` 三个只读 action，让 Agent 可主动复查，不增加第二套 API。
