# Mimi Mail 历史邮箱闭环调研报告

日期：2026-07-15
状态：已审核（持续授权）

## 调研范围

- `examples/connectors/macos-mail-connector.mjs`
- `tests/macos-mail-connector.test.ts`
- `mimi.connectors.example.json`
- `docs/CONNECTORS.md`

## 核心发现

Mail Connector 已能递归列出账号邮箱的精确 path，也能按该 path 移动收件箱邮件；但 `search_messages` 固定读取 `app.inbox().messages()`。归档项目、已发送和自定义目录无法检索。

直接遍历所有账号和全部邮箱会把一次查询扩大为不可控的历史扫描，容易触发 Mail JXA 超时。更合适的边界是：默认行为仍搜索统一收件箱；只有 Agent 先调用 `list_mailboxes` 获得账号和完整路径后，才把可选 `mailboxPath` 传给 `search_messages`，单次只查询一个明确邮箱且结果最多 100 条。

现有 `findMessage()` 同样固定在统一收件箱。若历史搜索只返回结果但后续 read/list attachments/save/reply/mark/flag/move/delete 无法定位该邮件，就不是事务闭环。需要一个可选 `source:{account,path}` locator，所有按 message ID 的动作复用；不提供时保持原收件箱兼容行为。

## 关键结论

- 扩展既有 `search_messages`，并提供只复用同一实现的 `search_mailbox_messages` 目录别名。原因是内置 catalog 升级只补缺失 action、不会覆盖 owner 已有描述；新 action 可让长期安装自动发现新参数和后续 source locator。
- `mailboxPath` 必须 1–20 段、每段 1–255 字符；使用路径时 target 必须是精确账号而非 `*`。
- 按 ID 的动作接受可选 `source` locator。JXA 使用现有逐层精确匹配和同层歧义拒绝逻辑，只读取该邮箱的 messages。
- 保持 limit/body/附件边界、argv JSON 传参和 action timeout；不做全局历史索引、邮箱镜像或自动目录猜测。
