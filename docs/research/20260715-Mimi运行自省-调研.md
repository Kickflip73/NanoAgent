# Mimi 运行自省调研报告

日期：2026-07-15
状态：已审核（持续授权）

## 调研范围

- 目标：让长期运行的 Mimi 能查看并主动汇报自身积压、失败和近期状态变化。
- 涉及文件：
  - `src/daemon/store.ts`
  - `src/daemon/dispatcher.ts`
  - `src/daemon/schedule-tools.ts`
  - `src/daemon/service.ts`
  - `src/daemon/cli.ts`
  - `src/runtime/tool-policy.ts`

## 核心发现

### 现状分析

Store 已持久化 Event、Run、Outbox、Digest、Schedule 和 Audit；CLI 能分别列出 events/runs/outbox，Daemon status 能返回计数。数据已经足够，不需要可观测性数据库、指标服务或新日志管线。

Agent 在 Daemon Event 中只获得 Schedule Host Tools 和可选 Connector Action Tool。即使是晨间规划、晚间收尾或 owner 直接询问，它也无法读取 Store 的积压、dead letter 和近期状态变化，因此无法主动回答“你最近做了什么、还有什么失败”。

### 关键流程

Dispatcher 每次执行 Event 时把 Host Tools 注入同一个 MimiAgent Run。只读自省工具可以直接查询同一个 `MimiStore`，不产生第二个 Agent、RPC 回环或并发写事务。CLI/RPC 也应复用同一个 Store 快照，避免两套统计定义。

### 现有约束

- 项目默认开放 Runtime 权限，不增加审批或角色等级。
- 长期运行数据必须有界；Host Tool 输出也必须有界。
- 外部 Event 同样拥有完整工具能力，因此自省输出只包含运行元数据、错误摘要和状态变化，不返回其他 Event 正文、Run 答案、Outbox payload 或 target。
- Audit 记录覆盖关键 Event 终态、retry/preempt、briefing 和 Outbox failure/repair，但不是完整业务日志；快照需要同时返回表状态与近期 transition，不能把 Audit 冒充完整历史。

## 与任务相关的关键结论

增加一个 Store `activitySnapshot(limit)` 即可统一生成：计数、是否需关注、待处理工作量、Digest/Schedule 数量、近期 Event/Run/Outbox 元数据和 Audit transition。Agent 使用一个 `inspect_mimi_activity` 只读 Host Tool，CLI/RPC 使用 `daemon activity [limit]` / `activity.get`。不新增表、迁移、线程、网络端点或审批模型。
