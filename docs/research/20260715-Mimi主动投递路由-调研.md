## 核心发现

### 现状分析

Connector 入站消息只有声明 `replyTarget` 时才生成 Event `replyRoute`。这适合 IM 原路回复，但 System、Calendar、Radar、Files 等主动来源通常没有会话目标。Dispatcher 对 `run` 结果仅在 Event 已有 `replyRoute` 时创建 Outbox，因此这些来源即使成功唤醒 Agent 并完成任务，也可能只把答案留在本机 Run 记录，owner 不会收到主动汇报。

`notify` 决策会回落到 system，但它只转发原始 Event payload，不包含 Agent 处理结果。Routine 和 Briefing 可配置 `replyChannel`，却不能配置 Connector 所需的 `target`。Agent 创建的 Schedule 会继承 Event route；无 route 时当前固定为 system，同样无法继承 owner 的首选 IM 会话。

### 可复用路径

- `ReplyRoute` 已是稳定的 `{channel,target?}` 类型，无需增加数据模型。
- Event 完成与单条 Outbox 插入已经在同一 SQLite 事务中，无需增加通知工作流。
- Connector `NotificationSink` 已按 `connector:<id>` 注册，可直接使用现有 target 投递。
- Attention 配置已支持热重载和严格 Zod 校验，适合保存一个 owner 默认 route。

## 关键结论

- 在 `owner` 下增加默认 `replyRoute`，缺省仍为 `{channel:"system"}`，旧配置无损加载。
- 来源 Event 自带 route 时始终优先，避免把 IM 对话回复错送到 owner 默认渠道；缺失时由 AttentionEngine 统一解析 owner route。
- Dispatcher 的 `run` 与 `notify`、Agent Schedule Tool 和 RPC Schedule 默认值都复用同一个解析结果。
- Routine/Briefing 保留 `replyChannel` 兼容字段并增加可选 `replyTarget`；未单独配置时继承 owner route。
- status 只暴露默认 channel，不返回 target；不增加 fan-out、路由规则 DSL、数据库表、线程或审批层。

## 风险与边界

- Connector channel 配置错误会沿用现有 Outbox 重试、dead letter 和 system fallback，不静默吞掉结果。
- 单路由意味着不会同时推送多个设备；这是保持事务语义和架构轻量的有意边界。
- 已经在旧配置中显式写入 `replyChannel:"system"` 的 Routine/Briefing 继续保持原行为，owner 可删除覆盖或补充 target。
