# Mimi 主动日常例程调研报告

日期：2026-07-15
状态：已审核

## 调研范围

- 目标：让 Mimi 即使没有新消息或摘要，也能按 owner 的日常节奏主动规划、检查和收尾。
- 涉及模块：`AttentionEngine`、`MimiDispatcher`、SQLite Event 去重、Schedule、Standing Orders。

## 核心发现

### 现状分析

主动简报只在摘要池存在待处理项目时创建 Event；空摘要日不会唤醒。持久 Schedule 支持一次绝对时间和固定毫秒 interval，适合 Agent 自建后续任务，但固定 interval 会随夏令时和停机时间漂移，不能清晰表达“本地时区每天 08:00”或“仅工作日 18:30”。

### 关键流程

AttentionEngine 已持有 owner timezone，并在每次 Dispatcher poll 时检查 briefing 时点。可在相同位置检查 daily routine：若当天星期匹配且本地时间已到，就生成 `attention:routine` owner schedule Event。现有 Event 唯一键 `(source, externalId)` 可用 `routine:<id>:<local-date>:<time>` 保证同一天只执行一次，Daemon 晚启动时也能补发当天已到时点。

### 现有约束

- Routine 只负责产生普通 Event，不直接调用模型或执行工具。
- 使用现有 Dispatcher、Attention、Session、Runtime、Connector Action Bridge、执行账本与 Outbox。
- 配置需要热重载、旧 `assistant.json` 兼容、数量与 prompt 总量上限。
- status 只展示条数与下一次语义，不返回私人 prompt 原文。

### 风险与问题

- 多条例程同一时刻会产生多个有界 Run，应限制总数并鼓励合并相关检查。
- Daemon 当天晚启动会补发已经到时的例程，这是可靠接管所需语义；如果 owner 不希望补发，可禁用或调整时间。
- Routine 是 owner 可信指令，默认开放工具；不应把外部消息正文写进 Routine 配置。

## 与任务相关的关键结论

无需 cron parser、调度表迁移或工作流引擎。给 Attention 配置增加最多 50 条 `id/time/weekdays/prompt/priority/sessionKey/replyChannel`，在 Dispatcher 的现有 poll 开头调用 `emitDueRoutines`，即可获得本地时区、热重载、跨重启幂等和稳定 Session 的主动个人运营节奏。
