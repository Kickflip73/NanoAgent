# Mimi 死信处置调研

## 现状

Event 与 Outbox 都有有界重试、dead letter、审计和最终失败通知。Event 还拥有按 Event ID + execution key 持久化的副作用执行台账，因此同一个 Event 再执行时可以跳过已经成功的确定性动作。

当前控制面只支持列出 Event、Run 与 Outbox。dead letter 一旦形成，只能被观察，无法恢复或明确标记为已处置。长期运行历史保留又会刻意永久保留 dead letter，最终会积累无法区分“仍需修复”和“owner 已知晓”的失败记录。

## 最小能力边界

- Event dead letter：允许显式重试或归档。重试保留原 Event ID，清理租约与本轮错误、从零开始新的有界尝试周期，从而继续复用既有 execution ledger。
- Outbox dead letter：允许显式重试或归档。重试保留原 Outbox ID；由于远端可能已经接收但本机未收到确认，命令结果必须明确提示可能重复投递。
- archived 是终态但不是活故障。现有历史保留任务可以在保留期后清理它，不需要新表、新线程或新审批模型。
- 后台不得自动重放 dead letter。只有 owner 明确执行 CLI/RPC 操作才改变状态。

## 一致性要求

四个状态变更都在 Store transaction 内完成，使用 `WHERE status = 'dead_letter'` 做 CAS，并写入 audit。非 dead letter、未知 ID 或并发状态变化都直接失败，不能静默成功。

Digest briefing 若被归档，与 dead letter 一样释放未消费摘要的 `briefing_event_id`，让后续窗口能够重新生成简报。

## 不做的事情

- 不增加角色、授权等级、审批单、确认提示或策略 DSL。
- 不复制 Event 或 Outbox，不改变 source/externalId 去重语义。
- 不自动判断外部投递是否成功，也不自动重放结果不确定的事务。
- 不引入独立 dead-letter queue 表；状态列已经足够。
