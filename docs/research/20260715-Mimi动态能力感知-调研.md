# Mimi 动态能力感知调研报告

日期：2026-07-15
状态：已审核（持续授权）

## 调研范围

- `src/daemon/connector-action-tool.ts`
- `src/daemon/connectors.ts`
- `src/daemon/dispatcher.ts`
- `src/runtime/tool-policy.ts`

## 核心发现

ConnectorManager 已能实时返回每个 Connector 的 enabled、online、source、trust 和 action 目录。`daemon connectors` 也能供人查看，因此不需要新状态表或健康服务。

Agent 当前只看到 `connector_action` 在 Run 开始时构造的静态 action 描述。描述没有 online/disabled 状态；Connector 在 Run 中退出、恢复或热重载后，模型也无法刷新能力视图。结果是 Mimi 可能盲目调用离线渠道，而不是选择当前在线的 Mail、Messages、系统通知或其他替代执行面。

`connectors.json` 的 connector/action record 数量没有 schema 上限，直接把完整目录放进 Tool description 或 Tool result 可能造成上下文膨胀。能力视图必须限制 Connector 数、总 action 数和描述长度，同时返回 total 与 truncated，不能假装是完整目录。

## 关键结论

- 从 `listCapabilities()` 派生一个无凭证、无命令路径的 bounded snapshot：总数、enabled/online/action 计数、最多 50 个 Connector 和全局最多 100 个 action。
- 新增 `inspect_mimi_capabilities` 只读 Tool，每次执行动态读取 Manager，因此能看到 Run 中状态变化。
- `connector_action` 的 Run 初始描述复用同一快照并标注 online/offline/disabled；执行前仍由 Manager 做最终状态检查。
- 事务超时或结果不确定后不自动重放。能力感知只帮助 Agent 选择不同执行面，不替代副作用台账或产生自动 fallback 引擎。
