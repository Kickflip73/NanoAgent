## 核心发现

### 大象官方边界

大象机器人事件订阅不是普通公网 HTTP Webhook。官方回调走主干环境的 Thrift/OCTO 服务，事件订阅需为 `com.sankuai.dxenterprise.open.gateway` 配置白名单，配置后必须发布应用。平台统一回调超时为 3 秒并最多重试 3 次，业务方应快速返回、异步处理并按消息 ID 幂等；Node 等非 Java 服务返回类型错误或 null 也会触发重复回调。

因此本机 MimiAgent 不能诚实地直接充当开放平台回调服务，也不应引入内网 SDK、OCTO 注册或公网监听。正确边界是：业务方已有的主干回调完成官方身份校验并立即返回，再把规范化事件通过受控通道转发到 Mimi 的 localhost Webhook。

### 现有中继缺口

Webhook 已具备 127.0.0.1 绑定、Bearer token、1MB 上限、60 次/分钟限制、严格 Zod schema、固定 external trust，以及 actor/conversation 字段。Store 使用 `source + externalId` 去重，正好能吸收大象重复回调。

但 Webhook 只能用 `notify:true` 指向 system，不能声明应由哪个 Connector 回到哪个会话。主干 relay 即使保留了大象 user/chat ID，Mimi 处理结果也无法原路回复。

上一阶段新增 owner fallback 后还暴露了一个兼容问题：`notify:false` 产生无 route Event，但 Dispatcher 会把它解释成“使用 owner route”，从而意外回传。需要区分普通来源的“未指定 route”与 Webhook 明确关闭通知。

## 关键结论

- Webhook 增加可选 `reply:{connector,target}`，connector 只允许稳定 ASCII ID，target 有界；Host 转换为 `connector:<id>` ReplyRoute。
- `reply` 优先于 `notify`；没有 reply 时，`notify:true` 保持 system，`notify:false` 保持无回传。
- Attention route resolver 对无 route 的 `webhook:*` Event 返回 undefined；其他自主来源仍继承 owner route。Dispatcher 的 run 结果据此选择是否创建 Outbox，notify 决策仍显式回落 system。
- actor/conversation/externalId 原样利用现有 schema 和 Store，不增加 Daxiang 专用核心类型、表、线程或依赖。
- 文档提供大象主干 relay 的规范化请求示例，但不伪造官方 Thrift IDL、验签或部署模板。

## 风险与边界

- localhost Webhook 不替代官方回调鉴权；Bearer token 只保护主干 relay 到本机的第二跳。
- 主干服务必须在自身可靠队列或异步执行器中转发，不能等待 Mimi 完成后才向大象返回。
- Connector 不在线时沿用现有 Outbox retry/dead-letter/system fallback；不自动重放 Agent 外部事务。
