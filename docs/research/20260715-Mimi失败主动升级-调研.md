# Mimi 失败主动升级调研报告

日期：2026-07-15
状态：已审核（owner 已明确要求按设计持续实施）

## 调研范围

- 目标：让无人值守任务和消息投递的最终失败主动触达 owner，而不是静默留在 dead letter。
- 涉及文件：
  - `src/daemon/store.ts`
  - `src/daemon/dispatcher.ts`
  - `src/daemon/notifier.ts`
  - `tests/daemon-store.test.ts`
  - `README.md`
  - `docs/ARCHITECTURE.md`

## 核心发现

### 现状分析

Event 在 `failEvent()` 达到最大尝试次数后会持久化为 `dead_letter`，Host Run 标为 failed；Outbox 在 `failOutbox()` 达到最大次数后也变成 `dead_letter`。两条路径都没有产生通知或后续 Event。Owner 只有主动执行 `daemon events` / `outbox` 才能发现，长期无人值守时失败可能无限期沉默。

Dispatcher 已经把 Outbox 作为可靠投递队列，并且 `system` channel 由本机 Notification Center/console 直接消费，不需要再次调用模型。因此失败升级不需要告警服务、第二个 Connector 或新的状态表。

### 事务与递归边界

- Event terminal failure 和 fallback Outbox 必须在同一个 SQLite transaction 中提交，避免崩溃后出现“已 dead letter 但没有通知”。
- 非 terminal retry 不应通知，避免短暂故障形成告警风暴。
- 非 system Outbox terminal failure 可以创建一个 system fallback，告诉 owner 原渠道未送达。
- system Outbox 自身 terminal failure 绝不能再生成 system fallback，否则会形成递归 dead letter。
- fallback 只引用原 Event ID，不创建新的 Agent Event，不触发 Attention/模型，也不增加权限模型。
- 通知正文只包含有界 event/outbox ID、source/channel、attempts 和错误摘要，不复制外部 payload、邮件正文或消息内容。

### 现有 schema 适配

`outbox.event_id` 必须引用现有 Event。任务失败通知可以直接绑定失败 Event；渠道 fallback 绑定原 Outbox 的同一 Event。现有 `insertOutbox()` 已在其他 Store transaction 中复用，schema 无需迁移。

`failOutbox()` 当前不是 transaction，也没有 audit。需要把状态更新、fallback 插入和 `outbox.retry/dead_letter` audit 合并为一个原子操作。

## 与任务相关的关键结论

在 `MimiStore` 的两个 terminal failure 分支中原子创建 system Outbox，是最小且可靠的主动升级机制。它复用现有重试、Outbox、Notifier 和本机系统通知，不增加配置 DSL、告警进程或审批步骤。
