## 核心发现

第 38 阶段保证所有有 route 的成功 Agent Run 都原子创建 Outbox，这解决了自主结果静默丢失。但周期 Schedule、Daily Routine 和低频状态复查即使确认“一切正常、没有变化”，也只能返回一段文本并投递。随着长期巡检增多，这会制造固定频率的无价值通知，违背“只在关键时刻主动唤醒”的产品定位。

模型不能可靠地用空字符串表达不通知：Dispatcher 仍会创建 Outbox，且文本 sentinel 会污染答案和协议。Attention 也无法提前决定，因为是否有变化通常要等 Agent 调用 Calendar、Mail、Radar 或其他工具后才知道。

Daemon 已支持每个 Event 注入闭包型 Host Tools。一个运行内 delivery control 可以在 Agent 完成事实检查后标记“成功但不投递”；Event result 和 Host Run 仍照常持久化，只有同事务内的 Outbox 省略。它不是外部副作用，不需要 execution ledger、表或 migration；失败、抢占或重试都会创建新的控制状态。

## 关键结论

- 新增 daemon-only `finish_mimi_silently(reason)` Host Tool，reason 1～500 字符，用于确认无变化、无需 owner 注意的自主运行。
- Tool 只注入非 `command` Event。CLI、IM 和 Webhook command 永远看不到它，从结构上保证直接请求有回答。
- Dispatcher 在成功提交时把 `{suppressed:true,reason}` 写入 Event result，并不创建 Outbox；答案和 usage 仍保留用于本机审计。
- Tool 属于 `control` capability，但不是 side effect；不进入 execution ledger，也不传给 SubAgent/Team worker。
- 默认晨间/晚间例程明确：有关键安排、已执行动作、风险或失败时汇报；确实无变化且无需关注时调用静默工具。
- 不增加阈值 DSL、通知评分器、第二模型或跨运行静默状态。

## 风险与边界

- 静默是模型在完成检查后的判断，不是形式化证明；Standing Orders 和 Routine prompt 应明确何时允许。
- action=notify 的 Attention 事件不运行模型，因此不会被静默工具抑制。
- Event 仍标记 completed，Activity/Audit 可证明巡检发生过；只是没有用户侧投递。
