# Mimi 开放权限内核精简调研报告

日期：2026-07-15  
状态：已审核

## 调研范围

- 目标：在默认开放所有事件能力后，删除不再进入执行链的 Approval/Mandate 复杂度。
- 涉及文件：`src/daemon/store.ts` `src/daemon/types.ts` `src/daemon/service.ts` `src/daemon/cli.ts` `src/runtime/tool-policy.ts` 及相关测试/文档。

## 核心发现

### 现状分析

Dispatcher 已不导入 Approval/Mandate 工具，所有来源都使用完整 Runtime 能力。但旧实现仍保留 3 个专用模块、多组类型、数百行 Store 方法、4 张 SQLite 表、RPC/CLI 入口和测试。这些入口会让用户以为仍需要审批，也使新数据库持续创建无用表。

### 迁移约束

- 新库应只创建 events、runs、outbox、leases、audit_events、schedules、digest_items 和 attention_state。
- 已有 v1–v4 库不删表、不删数据；只升级 `user_version=5`，老表成为未读取的历史数据。
- v1/v2 旧库仍需补建 digest/attention 表；v3/v4 可直接升级。
- 副作用语义执行台账、Connector action 能力目录和凭证隔离必须保留，它们是可靠性/适配器契约，不是权限审批。

## 关键结论

采用 schema v5 的“新库直接最小化、旧库非破坏性前进”策略。删除所有可调用 Approval/Mandate 代码和公开命令，不对旧表做 DROP，即可同时获得轻量代码和无损升级。
