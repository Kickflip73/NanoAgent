# Mimi 紧急事件抢占调研报告

日期：2026-07-15
状态：已审核（owner 已明确要求按设计持续实施）

## 调研范围

- 目标：确保长任务运行期间，关键告警和 owner 紧急命令仍能及时唤醒 Mimi。
- 涉及文件：
  - `src/daemon/dispatcher.ts`
  - `src/daemon/store.ts`
  - `src/daemon/attention.ts`
  - `src/runtime/run-service.ts`
  - `src/runtime/mimi-agent.ts`
  - `src/runtime/tool-ledger.ts`
  - `tests/daemon-store.test.ts`

## 核心发现

### 现状分析

Mimi 使用单 Dispatcher 和单 MimiAgent，这是轻量且能避免 Session 竞争的正确边界。但 `processEvent()` 会完整等待一次模型 Run，主循环在此期间既不 claim 新 Event，也不投递 Outbox。一个耗时数分钟的低优先级检查会让随后到达的 priority 100 owner 命令、告警或系统健康事件一直排队。

Store 的 `claimEvent()` 已按 `priority DESC, received_at ASC` 选择工作，因此阻塞只发生在当前 Run 内，而不是队列排序错误。Attention 的 `quietHours.urgentPriority` 已是 owner 可配置的紧急阈值，默认 95，可直接复用，不需要第二套优先级配置。

### 可复用的恢复语义

- MimiAgent/Agents SDK 已接受 AbortSignal；中止会以 interrupted Run 结束并释放 active Session。
- Daemon Event 使用稳定 `executionKey = event:<id>`，成功的 Schedule/Connector 等副作用保留在语义账本中。被抢占 Event 重跑时不会重复已经完成的相同外部动作。
- Store 已持久化 Host Run 和 Event lease，但普通 `failEvent()` 会增加退避并最终 dead letter。抢占不是任务失败，必须使用独立的 requeue 事务：撤销本次 claim 的 attempts、把 Host Run 标为 interrupted、立即恢复 queued。

### 抢占边界

- 只有 ready、priority 达到 owner `urgentPriority`、严格高于当前 Event，且经过现有 Attention 判定会 `run` 或 `notify` 的事件可以抢占。高 priority 但应 digest/ignore 的环境噪声不会制造中断，也不会遮住同批后续真实告警。
- 只在模型思考阶段中止 Run。通过既有 stream 的 `tool_called/tool_output` 维护在途计数；任何本地、MCP 或 Connector 工具执行中都暂缓抢占。Attention 分类、SQLite 事务和 Outbox/Connector delivery 同样不被打断，避免制造结果不确定的外部事务。
- 每个 Run 只创建一个短周期 timer 查询最高优先级 ready Event；Run 结束立即清理。没有第二 Dispatcher、并发 Agent、独立优先队列或常驻调度线程。
- 抢占原因记录目标 Event ID/priority，进入 Event error、Host Run 和 audit；原 Event 立即回队列，高优先级事件由既有排序先执行。
- 相同或更低优先级不能互相抢占，避免同等级消息形成抖动和饥饿。

## 与任务相关的关键结论

在单 Dispatcher 内为每次 Agent Run 增加一个有界 preemption monitor，是兼顾关键时刻响应和轻量架构的最小闭环。Store 增加只读 ready peek 与事务性 `preemptEvent()`，Attention 暴露已有 urgentPriority；无需数据库迁移或新服务。
