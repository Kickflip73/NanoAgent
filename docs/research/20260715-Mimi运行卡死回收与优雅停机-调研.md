# Mimi 运行卡死回收与优雅停机调研报告

日期：2026-07-15
状态：已审核（owner 已明确授权持续直接实施）

## 调研范围

- 目标：确认单 Dispatcher 在模型无响应、工具执行和 Daemon 停机时是否能可靠释放当前 Event。
- 涉及文件：
  - `src/daemon/dispatcher.ts`
  - `src/daemon/attention.ts`
  - `src/daemon/store.ts`
  - `src/runtime/run-service.ts`
  - `tests/daemon-dispatcher.test.ts`

## 核心发现

### 现状分析

Dispatcher 为运行中 Event 每隔约三分之一 lease 时长续租，因此正常长任务不会被其他 worker 误领。Agent Run 接收 AbortSignal，紧急事件可以在没有 Tool 执行时中止当前模型并无惩罚重排队。

当前没有模型无进展时限。若 provider 建连、流读取或模型思考永久不返回，Dispatcher 会持续续租，单执行器无法处理普通后续 Event。紧急抢占虽然会 abort，但只有出现更高优先级 Event 才触发，不能作为常规卡死恢复机制。

此外，`stop()` 会 abort Dispatcher 总 signal；`processEvent()` 当前把这条路径当普通错误交给 `failEvent()`，导致正常重启也增加 attempts、进入退避，极端情况下甚至 dead letter。

### 关键流程

1. `claimEvent()` 将 attempts 加一并建立 lease。
2. `processEvent()` 定时续租，然后调用 `AgentRunService.execute()`。
3. 只有紧急抢占拥有独立的无惩罚 `preemptEvent()` 路径。
4. 其他异常统一 `failEvent()`，包括 provider 错误和 Daemon 主动停止。

### 现有约束

- 单 Dispatcher、单 Agent、稳定 Session 写者必须保持，不能通过并发第二个 Agent 掩盖卡死。
- Tool 可能正在执行真实外部事务；不能仅因模型 watchdog 到期就中途 abort 并自动重放。
- Attention 配置已经支持原子读取、严格校验、热重载和隐私化 status，适合作为运行策略的唯一配置入口。
- Event lease、Run 状态和副作用 ledger 已有足够语义，不需要 schema 迁移。

### 风险与问题

- 只用硬总时长会误杀仍在持续产出进展的长任务。
- Tool 执行期间继续计算 idle timeout 会让结果不确定的外部事务面临重放风险。
- 仅 `Promise.race` 后抛弃底层 Agent Promise 会留下 Session owner 和迟到回调，不能安全使用。
- Abort 依赖 SDK/provider 遵守 signal；本阶段不伪装成独立进程级强杀。系统 supervisor 仍负责进程级故障恢复。

## 与任务相关的关键结论

增加“无 Agent 进展超时”而非硬总时长：模型流或 Runtime Event 到来时刷新计时；Tool 开始时暂停，最后一个 Tool 输出后恢复。超时通过现有 AbortSignal 进入普通失败重试，保留 attempts/dead-letter/最终主动升级语义。Daemon 总 signal 的中断则复用 `preemptEvent()` 无惩罚重排队。配置放入 `assistant.json` 并可热重载，默认 20 分钟，不增加新服务、表或权限模型。
