# Mimi 连接器健康监控调研报告

日期：2026-07-15
状态：已审核

## 调研范围

- 目标：让长期运行的 Mimi 在外部信息源失联和恢复时主动感知并通知所有者，同时避免反复崩溃造成事件风暴。
- 涉及文件：
  - `src/daemon/connectors.ts`
  - `src/daemon/store.ts`
  - `src/daemon/attention.ts`
  - `tests/daemon-connectors.test.ts`
  - `docs/CONNECTORS.md`

## 核心发现

### 现状分析

Connector Host 已经负责独立子进程拉起、最长 60 秒指数退避重启、在线能力查询以及断线时拒绝未确认的 delivery/action。故障信息目前只写入 daemon stderr；Store、Attention 与系统通知均不知道某个信息源已经失联。

### 关键流程

外部事件通过 Connector 直接进入 SQLite Inbox，随后由既有 Dispatcher、Attention Engine、Agent 与 Outbox 处理。因此健康变化也可以表示为普通内部 Event，不需要新表、新队列或第二套监控执行器。

### 现有约束

- Connector 的外部副作用在断线或超时后不得自动重放。
- Daemon 必须先持久化事件再执行，并继续使用统一租约、重试和通知链路。
- 架构应保持零新增依赖；连接器凭证和平台逻辑继续隔离在子进程。
- `trust` 只记录来源，不作为权限审批边界。

### 风险与问题

- 子进程成功 `spawn` 不等于已经稳定恢复；启动即崩的循环若每次都发“恢复/离线”会形成通知风暴。
- 正常 daemon 停止不应产生离线告警。
- 初次启动失败也必须可见，否则错误命令、系统权限或损坏配置会长期静默失效。
- 恢复提醒不应绕过静默时段和用户注意力预算。

## 与任务相关的关键结论

在 `ConnectorProcess` 内增加三态健康状态（初始、故障打开、稳定健康）即可覆盖需求。首次异常退出只产生一个高优先级内部告警；自动重启期间保持同一个故障窗口，只有子进程持续存活超过稳定阈值才产生恢复事件并关闭窗口。事件继续走现有 Store 与 Attention：故障可及时判断，恢复可按静默时段或预算进入摘要。该方案不新增持久化模型、监控服务或权限层。
