# Mimi 认证回调中继实施计划

日期：2026-07-15
状态：已完成
关联调研：`20260715-Mimi认证回调中继-调研.md`

## 任务目标

让大象等必须经过官方服务端回调的渠道，在安全转发到本机 Mimi 后保留人物、会话和可靠原路回复能力，同时恢复 `notify:false` 的无回传语义。

## 方案概述

扩展既有 localhost Webhook 的外部 schema，不新建端口或服务。relay 提交 connector ID 和渠道 target，Webhook 转换成现有 ReplyRoute；Dispatcher 继续使用单 Event、单 Agent、单 Outbox 事务。

## UI 变动检测

涉及 UI 变动：否
变动类型：无
预览状态：不适用

## 详细步骤

1. 为 `/v1/events` 增加严格 `reply.connector/target` schema；reply 优先，notify 保持兼容。
2. 让 Attention route resolver 对显式无通知的 `webhook:*` 保留 undefined，Dispatcher run 结果按可选 route 创建 Outbox。
3. 测试认证、固定 external trust、actor/conversation、reply route、重复 externalId、notify=false、非法 connector/target 和 owner fallback 不回归。
4. 更新 Connector/Attention/Architecture/README/CHANGELOG 和总路线图，记录大象 Thrift/OCTO 的真实部署边界。
5. 运行聚焦检查和完整 CI。

## 权衡与考量

- 使用协议中立 relay，不把大象内网 SDK耦合到本地产品。
- reply 只能选择已配置 Connector ID，不能提交任意 channel 字符串。
- 不新增回调任务表；Store 的 Event 唯一键和 Outbox 已覆盖幂等与可靠性。
