# Mimi 跨渠道人物上下文实施计划

日期：2026-07-15
状态：已完成
关联调研：[[20260715-Mimi跨渠道人物上下文-调研.md]]

## 任务目标

让 owner 明确配置的同一人物在不同 Connector/会话来源中共享稳定 Session、替身决策上下文与长期记忆召回。

## 方案概述

扩展现有 `assistant.json` 为有界 `people` 数组。AttentionEngine 以 `source + actor.id` 确定性解析 canonical person，再复用现有 Policy、RunCause、Session、Memory 和 Context 链路。所有能力仍由同一个 Agent 执行。

## UI 变动检测

涉及 UI 变动：否
变动类型：无
涉及文件：无
预览状态：不适用

## 详细步骤

### 1. People 配置与解析

**涉及文件：** `src/daemon/attention.ts`

增加稳定 person ID、displayName、aliases 和可信 context；限制人物、alias、指令数量与字符总量，拒绝重复 ID/精确 alias。旧配置默认空数组，热重载沿用现有路径。

### 2. Person Session 与决策上下文

**涉及文件：** `src/daemon/policy.ts`、`src/runtime/mimi-agent.ts`

显式 `sessionKey` 继续优先；否则已解析人物使用 `mimi-person-<id>`。人物映射与 context 作为 owner 配置可信分区进入 input，外部正文仍是数据。RunCause 增加 person metadata。

### 3. 人物感知 Memory

**涉及文件：** `src/core/memory.ts`、`src/core/context.ts`、`src/runtime/mimi-agent.ts`

自主记忆附带 person、actor、conversation；检索 query 和索引文本包含 provenance；上下文展示人物标签，跨来源消息可以召回同一人的既有偏好与承诺。

### 4. 测试与文档

**涉及文件：** `tests/daemon-attention.test.ts`、`tests/daemon-policy.test.ts`、`tests/core.test.ts`、`docs/ATTENTION.md`、`docs/ARCHITECTURE.md`、`README.md`、`SECURITY.md`、`CHANGELOG.md`

覆盖解析顺序、跨渠道稳定 Session、显式 Session 优先、可信/不可信分区、旧配置、热重载、status 隐私、Memory 跨渠道人物召回和上限，再运行完整 CI。

## 权衡与考量

- 不做自动身份推断；只有 owner 明确 alias 才合并。
- 不同步 Contacts 数据；Contacts Connector 继续负责按需查询。
- 不增加 Person Store；配置只影响现有事件上下文和 Memory metadata。
