# Mimi 紧急事件抢占实施计划

日期：2026-07-15
状态：已完成
关联调研：[[20260715-Mimi紧急事件抢占-调研.md]]

## 任务目标

让运行中的低优先级 Agent 任务在关键告警或 owner 紧急命令到达时安全让路，并在紧急事件处理后继续执行，避免长期任务阻塞关键响应。

## 方案概述

继续保持单 Dispatcher、单 MimiAgent。Agent Run 期间用一个短周期 timer 查询 ready 队列；达到 Attention 既有 urgentPriority 且严格高于当前 Event 时，中止本轮 Run。Store 以独立事务把当前 Event 无失败惩罚地回队列并把 Host Run 标为 interrupted，既有优先级排序随后先 claim 紧急事件。

## 详细步骤

### 1. Store 抢占原语

**涉及文件：** `src/daemon/store.ts`

新增有界 ready Event 只读查询，按阈值和当前优先级筛选。Dispatcher 再复用 Attention 过滤 digest/ignore 噪声。新增 `preemptEvent()` 原子恢复 queued、撤销本次 claim attempts、释放 lease、记录原因，并将关联 Host Run 标为 interrupted。

### 2. Attention 紧急阈值复用

**涉及文件：** `src/daemon/attention.ts`

只读暴露当前配置中的 `quietHours.urgentPriority`，reload 后自动生效，不增加第二个配置字段。

### 3. 单 Run 抢占监视

**涉及文件：** `src/daemon/dispatcher.ts`

在真正调用 AgentRunService 前创建局部 AbortController，并与 Daemon stop signal 合并。复用 stream observer 统计 `tool_called/tool_output`；工具在途时不抢占。timer 在模型思考阶段发现更高优先级紧急 Event 时记录目标并 abort；catch 分流到 `preemptEvent()`，其他错误继续走既有 bounded retry。所有终态清理 timer。

关键逻辑：

```ts
const urgent = store.readyEventsAbove(attention.urgentPriority, current.priority)
  .find((event) => ['run', 'notify'].includes(attention.decide(event).action));
if (urgent) runController.abort(new Error(`被紧急事件 ${urgent.id} 抢占`));
```

### 4. 确定性测试与文档

**涉及文件：** `tests/daemon-store.test.ts`、`tests/daemon-dispatcher.test.ts`、`README.md`、`docs/ARCHITECTURE.md`、`docs/ATTENTION.md`、`SECURITY.md`、`CHANGELOG.md`

验证阈值/严格高于/ready 边界、attempts 无惩罚、Host Run interrupted、低任务→紧急任务→低任务续跑顺序、相同优先级不抖动，以及完整 CI。

## 权衡与考量

- 不并发运行第二个 Agent，保留 Session、工具与外部副作用的单写者模型。
- 不抢占任何在途 Tool、Outbox 或 Connector 请求，宁可让关键事件等待一次有界调用，也不猜测外部事务结果。
- 抢占只针对严格更高优先级，紧急消息之间保持 FIFO。
- 原 Event 使用相同 executionKey 续跑，已成功副作用由现有 ledger 去重。
