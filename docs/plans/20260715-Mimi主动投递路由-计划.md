# Mimi 主动投递路由实施计划

日期：2026-07-15
状态：已完成
关联调研：`20260715-Mimi主动投递路由-调研.md`

## 任务目标

让所有没有来源回信地址的自主事件，在 Agent 处理完成后可靠推送给 owner，并让例程、简报和后续计划能够投递到具体 Connector 会话。

## 方案概述

复用现有 `ReplyRoute`、Attention 配置和单 Outbox 事务。在 AttentionEngine 提供唯一的 route resolver：Event route 优先，否则使用 owner 默认 route。Routine/Briefing 的兼容覆盖也在创建 Event 时解析为同一类型。

## UI 变动检测

涉及 UI 变动：否
变动类型：无
预览状态：不适用

## 详细步骤

1. 为 owner 增加严格校验、热重载兼容的 `replyRoute`，默认 system；状态只报告 channel。
2. Routine/Briefing 增加可选 `replyTarget`，并在未覆盖 channel 时继承 owner route。
3. Dispatcher 对 `run` 和 `notify` 始终使用 Event route 或 owner fallback，继续原子提交单条 Outbox。
4. Schedule Tool 与 RPC 新建 Schedule 继承同一已解析 route。
5. 覆盖来源优先、owner fallback、target 隐私、旧配置、热重载、Routine/Briefing/Schedule 和投递测试。
6. 更新 Attention/Architecture/README/CHANGELOG、总路线图和阶段记录，运行完整 CI。

## 权衡与考量

- 单一默认路由足以打通主动推送，不增加多路 fan-out 的部分成功语义。
- 继续使用已有 Outbox retry/dead-letter/system fallback，不建立第二套通知可靠性模型。
- target 属于本机私人路由信息，不进入 status 输出。
