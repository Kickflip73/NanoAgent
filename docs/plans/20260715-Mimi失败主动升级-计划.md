# Mimi 失败主动升级实施计划

日期：2026-07-15
状态：已完成
关联调研：[[20260715-Mimi失败主动升级-调研.md]]

## 任务目标

让 Mimi 在任务最终失败或非系统渠道最终无法投递时，可靠主动通知 owner，并保证升级通知不重复、不递归、不泄露原始外部内容。

## 方案概述

只修改现有 Store 终态事务。Event dead letter 时插入 system Outbox；非 system Outbox dead letter 时插入 system fallback。system fallback 若自身失败则停止。所有通知使用有界元数据摘要并复用现有 Notifier，不新增 Event、表、服务或权限层。

## UI 变动检测

涉及 UI 变动：否
变动类型：后台可靠性逻辑
涉及文件：无前端文件
预览状态：用户要求跳过

## 详细步骤

### 1. 有界失败通知载荷

**涉及文件：** `src/daemon/store.ts`

增加纯函数规范化错误摘要，并分别构造 `event_dead_letter` / `delivery_dead_letter` payload。正文限制在系统通知上限内，不带 Event 原始 payload 或目标正文。

### 2. Event terminal 原子升级

**涉及文件：** `src/daemon/store.ts`

在既有 `failEvent()` transaction 内，仅 terminal 分支调用 `insertOutbox(event.id, {channel:'system'}, ...)`。非 terminal retry 行为保持不变。

### 3. Outbox terminal fallback

**涉及文件：** `src/daemon/store.ts`

把 `failOutbox()` 改为 transaction，并校验 lease 更新成功。terminal 且原 channel 非 system 时插入 system fallback；system dead letter 不再递归。补充 retry/dead-letter audit。

关键逻辑：

```ts
if (terminal && message.channel !== 'system') {
  insertOutbox(message.eventId, { channel: 'system' }, deliveryFailure(message, error), now);
}
```

### 4. 确定性测试与文档

**涉及文件：** `tests/daemon-store.test.ts`、`README.md`、`docs/ARCHITECTURE.md`、`SECURITY.md`、`CHANGELOG.md`

验证 retry 不通知、Event terminal 一次通知、Connector Outbox terminal 一次 fallback、system fallback terminal 不递归、payload 无原正文，以及完整 CI。

## 权衡与考量

- 直接系统通知比再生成 Agent Event 更可靠：Provider 失败正是 Event dead letter 的常见原因。
- 只在最终失败升级，避免瞬时网络错误制造噪声。
- system channel 失败后只能留给 `daemon outbox`/日志检查；继续自动升级没有独立可靠通道，只会递归。
- 不自动重放结果不确定的外部事务，通知 owner 不改变既有 at-most-once 边界。
