# Mimi 长期运行历史保留实施计划

日期：2026-07-15
状态：已完成
关联调研：[[20260715-Mimi长期运行历史保留-调研.md]]

## 任务目标

让 Mimi 的长期 SQLite 控制面在持续接收工作和生活事件时保持有界，同时不删除任何待执行、待投递、待简报或最终失败证据。

## 方案概述

给现有 `assistant.json` 增加 `maintenance.enabled/historyRetentionDays/intervalHours`。Store 提供一个事务性 `pruneHistory(cutoff)`，按外键依赖顺序清理安全终态；Dispatcher 复用自身循环低频触发。无新 schema、线程、服务或定时器。

## 详细步骤

### 1. 可热重载维护配置

**涉及文件：** `src/daemon/attention.ts`

```ts
maintenance: {
  enabled: true,
  historyRetentionDays: 90,
  intervalHours: 24
}
```

保留期限制 7 至 3650 天，检查周期限制 1 至 168 小时；旧配置自动采用默认值，status 只显示数值。

### 2. 引用安全的 Store 清理

**涉及文件：** `src/daemon/store.ts`

在同一 transaction 中按依赖顺序删除旧 sent Outbox、已归档 Digest、候选 Event 的 Runs 和无引用终态 Event；再清理旧 disabled Schedule、Attention checkpoint 与普通 Audit。dead letter 和所有活状态排除。返回各表删除计数，事务后执行 `PRAGMA optimize` 与 passive WAL checkpoint。

### 3. 复用 Dispatcher 周期

**涉及文件：** `src/daemon/dispatcher.ts`

`processOnce()` 开头检查内存中的下次维护时点。到期后从最新 Attention 配置计算 cutoff 并调用 Store；失败只写 stderr，不阻断 Event/Outbox。禁用时不清理。

### 4. 确定性测试与文档

**涉及文件：** `tests/daemon-store.test.ts`、`tests/daemon-attention.test.ts`、`tests/daemon-dispatcher.test.ts`、`README.md`、`docs/ARCHITECTURE.md`、`docs/ATTENTION.md`、`SECURITY.md`、`CHANGELOG.md`

验证清理安全终态、保留 active/dead-letter/未归档引用、默认值/旧配置/热重载、禁用和低频调用，并运行完整 CI。

## 权衡与考量

- 90 天 dedupe 窗口之后，极旧来源项若再次被 Connector 回放可能重新处理；换取控制面长期有界。
- dead letter 默认永久保留，owner 修复后未来可提供显式归档，不在自动维护中猜测。
- 不自动 `VACUUM`，避免常驻服务长时间独占锁；删除后的页面会被 SQLite 后续写入复用。
- 不改变默认开放 Runtime 权限或事件来源语义。
